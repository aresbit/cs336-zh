# 第1章 分词 (Tokenization)

> 译自 CS 336 Language Modeling from Scratch 课程讲义 (Tatsu Hashimoto)，并补充算法推导与领域知识。
> 分词是语言模型的第一道工序：它将原始文本转化为模型可以处理的整数序列。本章从 BPE 的完整推导出发，覆盖 SentencePiece/Unigram、训练与推理流程、词表大小权衡及多语言分词陷阱。

---

## 1.1 为什么分词如此重要？

大语言模型（LLM）在数学上是定义在**离散符号**上的概率模型。Transformer 的输入不是 Unicode 字符串，而是一个整数向量：

$$
\mathbf{x} \in \{0, 1, \ldots, |V|-1\}^{L}
$$

其中 $$|V|$$ 是词表大小，$$L$$ 是序列长度。分词器（tokenizer）负责在"人类可读的文本"和"模型可消费的整数序列"之间建立双向映射：

$$
\text{encode}: \Sigma^* \to \{0, \ldots, |V|-1\}^*
$$

$$
\text{decode}: \{0, \ldots, |V|-1\}^* \to \Sigma^*
$$

其中 $$\Sigma$$ 是字符集（通常为 Unicode），$$\Sigma^*$$ 是所有可能字符串的集合。分词器的设计直接影响以下几个方面：

1. **模型参数量**：词表大小 $$|V|$$ 决定了 LM head（输出层）的参数量为 $$|V| \times d_{\text{model}}$$。例如 GPT-3 的 $$|V| = 50257$$，仅输出嵌入矩阵就占用 $$50257 \times 12288 \times 2 \text{ bytes} \approx 1.2\text{ GB}$$（半精度）。
2. **序列长度**：分词粒度越粗，每个 token 承载的信息量越大，同一段文本产生的 token 数越少——这意味着相同的上下文窗口覆盖更多的语义信息。
3. **跨语言公平性**：不同语言的分词效率差异显著——中文每个 token 通常约 1.5 个汉字，而英文可达 4-5 个字符。在相同的 token 预算下，中文处于劣势。
4. **OOV (Out-Of-Vocabulary) 问题**：任何固定词表都不可避免地遇到未登录词。好的分词策略必须在词表大小和 OOV 比例之间取得平衡。

现代 LLM 的分词系统几乎全部基于**子词（subword）分词**——将词分解为更小的有意义的片段。这一思想是 BPE、SentencePiece 和 Unigram 等算法的共同起点。

## 1.2 Byte-Pair Encoding (BPE)：完整算法推导

### 1.2.1 问题设定

给定一个语料库（表示为 Unicode 字符序列），BPE 的目标是学习一个大小为 $$|V|$$ 的子词词表。设初始词表 $$V^{(0)}$$ 为语料中出现的所有**字节**（或字符，取决于实现），即 $$V^{(0)} = \{b \in \{0,\ldots,255\}\}$$（字节级 BPE）或 $$V^{(0)} = \{c \in \Sigma \mid c \text{ 出现在语料中}\}$$（字符级 BPE）。

算法通过 $$|V| - |V^{(0)}|$$ 轮**合并操作**将词表从 $$V^{(0)}$$ 逐步扩展到目标大小。

### 1.2.2 核心数据结构

BPE 的每一次迭代需要回答：语料中哪个相邻符号对 $$(a, b)$$ 出现的频率最高？朴素的做法（每轮重新扫描整个语料）是 $$O(N|V|)$$ 复杂度，对于现代语料不可接受。高效的 BPE 实现维护以下数据结构：

**符号对计数表**：对所有相邻对 $$(a, b)$$ 在语料中的出现频次进行索引。形式化地，设语料的符号序列为 $$s_1, s_2, \ldots, s_N$$（$$s_i \in V^{(t)}$$），定义：

$$
\text{count}(a, b) = |\{i \mid s_i = a \land s_{i+1} = b, \; 1 \leq i < N\}|
$$

设当前最大频率的对为 $$(a^*, b^*)$$，其频次为 $$f^* = \max_{a,b} \text{count}(a, b)$$。合并操作为：

$$
\text{merge}(a^*, b^*) \to c_{\text{new}}
$$

其中 $$c_{\text{new}}$$ 是一个新符号（通常表示为 $$a^*$$ 和 $$b^*$$ 的字符串连接）。合并后，词表更新为：

$$
V^{(t+1)} = V^{(t)} \cup \{c_{\text{new}}\}
$$

**关键优化**：合并只改变包含 $$(a^*, b^*)$$ 的位置。具体而言，对于语料中每个出现模式 $$x \; a^* \; b^* \; y$$ 的位置，将其替换为 $$x \; c_{\text{new}} \; y$$，然后更新以下计数：

- 删除：$$\text{count}(a^*, b^*) \gets 0$$（该对已被合并并消失）
- 删除：$$\text{count}(x, a^*)$$ 减少该位置处 $$(x, a^*)$$ 原先的计数
- 删除：$$\text{count}(b^*, y)$$ 减少该位置处 $$(b^*, y)$$ 原先的计数
- 新增：$$\text{count}(x, c_{\text{new}})$$ 增加
- 新增：$$\text{count}(c_{\text{new}}, y)$$ 增加

这里的关键洞察是：一次合并只影响与新符号 $$c_{\text{new}}$$ 相邻的对，而不会影响语料中其他位置的计数。这使得每轮合并的复杂度与语料大小成线性关系。

### 1.2.3 算法伪代码

```
输入: 语料 C, 目标词表大小 vocab_size
输出: 合并规则列表 merges

1. 将语料转换为字节序列 (或 Unicode 字符序列)
2. V = 所有出现在语料中的单字节 (或单字符)
3. 将语料中的每个词表示为符号序列, 词之间用特殊符号 (如 </w>) 分隔

4. while |V| < vocab_size:
    a. 扫描语料中的所有相邻符号对, 统计频次
    b. 找出频率最高的对 (a*, b*);  若有平局, 按某种确定性规则选择
    c. 创建新符号 c_new = a* + b*
    d. V = V ∪ {c_new}
    e. merges.append( (a*, b*) → c_new )
    f. 在语料中替换所有 (a*, b*) 为 c_new

5. 返回 V, merges
```

### 1.2.4 合并规则的正确性

BPE 的核心性质：合并规则构成一棵**合并二叉树**。词表中的每个符号要么是原子符号（叶子），要么由两个更小的子符号合并而来。这意味着任意 token 都可以被分解为更小的 token 序列——BPE 天然支持对未登录词的分解。

形式化地，设 $$\mathcal{M} = [(a_1, b_1), (a_2, b_2), \ldots, (a_K, b_K)]$$ 为训练阶段产生的 $$K = |V| - |V^{(0)}|$$ 条合并规则。在**编码（encode）**阶段，给定一个新词 $$w$$，BPE 执行以下操作：

1. 将 $$w$$ 分解为原子符号序列 $$w = [c_1, c_2, \ldots, c_m]$$
2. 按顺序遍历合并规则列表 $$\mathcal{M}$$：
   - 对规则 $$(a_k, b_k)$$，在当前序列中查找所有连续的 $$a_k, b_k$$ 对
   - 将每个匹配替换为合并后的符号
3. 返回最终序列

### 1.2.5 BPE 编码示例：从字符到有意义的子词

考虑一个极小的英文语料，字符集为 `{l, o, w, e, r, _}`（其中 `_` 代表空格）。语料包含以下 tokenized 词（初始状态）：

```
_ l o w
_ l o w e r
_ l o w e s t
_ n e w e r
_ n e w e s t
_ w i d e r
_ w i d e s t
```

设初始词表 $$V^{(0)} = \{l, o, w, e, r, \_, n, s, t, i, d\}$$，共 11 个符号。目标词表大小 $$|V| = 16$$。

**第 1 轮合并**：统计所有相邻对频次。最高频率对为 `(l, o)` 和 `(e, r)`，假设选择 `(e, r)` 合并为 `er`。

$$V^{(1)} = V^{(0)} \cup \{\text{er}\}$$

语料更新为：

```
_ l o w
_ l o w er
_ l o w e s t
_ n e w er
_ n e w e s t
_ w i d er
_ w i d e s t
```

**第 2 轮**：频率最高对为 `(o, w)`，合并为 `ow`。

$$V^{(2)} = V^{(1)} \cup \{\text{ow}\}$$

语料更新为：

```
_ l ow
_ l ow er
_ l ow e s t
_ n e w er
_ n e w e s t
_ w i d er
_ w i d e s t
```

**第 3 轮**：最高频对为 `(l, ow)`，合并为 `low`。

$$V^{(3)} = V^{(2)} \cup \{\text{low}\}$$

语料更新为：

```
_ low
_ low er
_ low e s t
_ n e w er
_ n e w e s t
_ w i d er
_ w i d e s t
```

**第 4 轮**：最高频对为 `(ew, er)`，合并为 `ewer`。

**第 5 轮**：最高频对为 `(d, er)`，合并为 `der`。

经过 5 轮合并，最终词表包含 16 个符号：`{l, o, w, e, r, _, n, s, t, i, d, er, ow, low, ewer, der}`。

注意：在 BPE 训练中，合并规则是由**语料整体统计**决定的，而非逐词进行的。如果 `low` 和 `lower` 在语料中高频出现，则合并 `l` 和 `ow` 形成 `low` 是有意义的——这与"子词应捕捉有意义的语言单元"的直觉一致。

## 1.3 BPE 的 Python 实现

以下给出基于字节级 BPE 的完整 Python 实现，涵盖训练、编码和解码：

```python
from collections import Counter, defaultdict
from typing import List, Tuple, Dict

class BPETokenizer:
    """字节级 BPE 分词器 (Byte-level BPE Tokenizer)"""

    def __init__(self, vocab_size: int = 256):
        """
        Args:
            vocab_size: 目标词表大小 (默认 256 对应所有单字节作为初始词表)
        """
        self.vocab_size = vocab_size
        self.merges: Dict[Tuple[int, int], int] = {}  # (a, b) -> new_id
        self.vocab: Dict[int, bytes] = {}              # id -> byte sequence

        # 初始化词表: 0-255 对应单个字节
        for i in range(256):
            self.vocab[i] = bytes([i])

    def _get_stats(self, ids: List[int]) -> Dict[Tuple[int, int], int]:
        """统计序列中所有相邻符号对的频次。

        遍历序列一次，用哈希表维护每对的出现次数。
        时间复杂度 O(n)，空间复杂度 O(|V|^3)（最坏情况，实际远小于此）。
        """
        counts = Counter()
        for pair in zip(ids, ids[1:]):
            counts[pair] += 1
        return counts

    def _merge(self, ids: List[int], pair: Tuple[int, int], new_id: int) -> List[int]:
        """将序列中所有出现的指定对替换为新 ID。

        关键实现细节: 使用双指针/while 循环避免 Python 切片开销。
        替换后跳过已处理位置以避免连锁合并。
        """
        new_ids = []
        i = 0
        while i < len(ids):
            # 检查当前和下一个位置是否匹配目标对
            if i < len(ids) - 1 and ids[i] == pair[0] and ids[i + 1] == pair[1]:
                new_ids.append(new_id)
                i += 2  # 跳过合并的两个位置
            else:
                new_ids.append(ids[i])
                i += 1
        return new_ids

    def train(self, text: str):
        """在给定文本上训练 BPE 分词器。

        训练流程:
        1. 将文本编码为 UTF-8 字节序列，每个字节作为一个初始 token
        2. 迭代合并在语料中频次最高的相邻对
        3. 记录合并规则 (用于后续编码)
        4. 更新词表 (ID -> 对应的字节序列)

        Args:
            text: 训练语料 (原始字符串)
        """
        # 步骤 1: 将文本编码为字节序列
        text_bytes = text.encode('utf-8')
        ids = list(text_bytes)  # 每个字节是一个 token ID

        # 步骤 2-3: 迭代合并
        num_merges = self.vocab_size - 256
        for i in range(num_merges):
            stats = self._get_stats(ids)
            if not stats:
                break  # 没有可合并的对了

            # 找频率最高的对 (平局时按字典序选择，保证确定性)
            best_pair = max(stats, key=lambda p: (stats[p], p))
            new_id = 256 + i

            # 记录合并规则
            self.merges[best_pair] = new_id

            # 更新词表: 新符号 = 旧符号的字节拼接
            self.vocab[new_id] = self.vocab[best_pair[0]] + self.vocab[best_pair[1]]

            # 在语料中应用合并
            ids = self._merge(ids, best_pair, new_id)

    def encode(self, text: str) -> List[int]:
        """将文本编码为 BPE token ID 序列。

        编码过程:
        1. 将文本转为字节序列
        2. 反复应用合并规则 (按训练时的顺序)
        3. 当没有规则可应用时返回最终 ID 序列

        这里的实现是贪心的: 按照合并规则的训练顺序依次尝试。
        现代 BPE 实现 (如 GPT tokenizer) 使用更高效的 trie 结构。

        Args:
            text: 要编码的文本

        Returns:
            token ID 列表
        """
        text_bytes = text.encode('utf-8')
        ids = list(text_bytes)

        # 按训练时的顺序应用所有合并规则
        # 对每一对 (a, b)，在当前序列中查找并合并所有出现
        while len(ids) >= 2:
            stats = self._get_stats(ids)
            # 查找训练集中存在且当前序列中也存在的合并规则
            pair = min(stats, key=lambda p: self.merges.get(p, float('inf')))
            if pair not in self.merges:
                break  # 没有更多可应用的合并规则
            ids = self._merge(ids, pair, self.merges[pair])
        return ids

    def decode(self, ids: List[int]) -> str:
        """将 token ID 序列解码回文本。

        解码相对简单: 查询每个 ID 对应的字节序列，拼接后用 UTF-8 解码。
        这是 BPE 的一个优势——解码不依赖任何启发式或语言模型。

        Args:
            ids: token ID 列表

        Returns:
            解码后的文本
        """
        # 拼接所有 ID 对应的字节序列
        text_bytes = b''.join(self.vocab[id] for id in ids)
        # UTF-8 解码 (忽略无法解码的字节)
        return text_bytes.decode('utf-8', errors='replace')


# ========== 使用示例 ==========
if __name__ == '__main__':
    # 小型语料演示
    corpus = (
        "low lower lowest new newer newest wider widest "
        "low lower lowest new newer newest wider widest "
        "the cat sat on the mat "
    )

    tokenizer = BPETokenizer(vocab_size=300)
    tokenizer.train(corpus)

    # 测试编码/解码
    test_text = "lowest"
    encoded = tokenizer.encode(test_text)
    decoded = tokenizer.decode(encoded)
    print(f"Input: {test_text}")
    print(f"Encoded ({len(encoded)} tokens): {encoded}")
    print(f"Decoded: {decoded}")
    print(f"Roundtrip OK: {test_text == decoded}")
    print(f"Vocab size: {len(tokenizer.vocab)}")
```

### 1.3.1 实现要点解析

**字节级 vs 字符级**：GPT-2 及之后的模型采用字节级 BPE——以 UTF-8 字节（而非 Unicode 字符）为初始词表。这保证了词表可以将**任意** UTF-8 字符串表示为 token 序列（因为每个字节都在初始词表中），从根本上消除了 OOV 问题。代价是某些非英文文本（尤其是 CJK 字符）可能被切成很长的字节序列，降低了编码效率。

**贪心编码的正确性**：在推理阶段，BPE 编码按照训练时合并规则的**创建顺序**依次应用。这种贪心策略保证了编码结果与训练过程一致——因为如果词 $$w$$ 在训练语料中出现过，其最终表示就是经过所有合并规则处理后的结果。

**合并顺序的优先级**：当多个合并规则可以应用到当前位置时，训练顺序（即频次高低）决定了优先级。先创建的合并规则（对应更高频的对）后应用——这是因为在推理阶段，我们按训练时的合并顺序扫描，高频规则在序列中的多个位置可能同时触发。

## 1.4 SentencePiece 与 Unigram 语言模型

### 1.4.1 SentencePiece 的设计哲学

SentencePiece（Kudo & Richardson, 2018）是一个语言无关的子词分词框架，被 LLaMA、T5、PaLM 等模型广泛使用。其核心理念是：**将输入视为原始字节流，而不仅仅是空格分隔的单词序列**。

传统分词器（包括 BPE）通常假设文本可以被预先切分为"词"（通过空格或特定语言的规则），然后在词内进行子词分割。这种假设在以下场景中会失效：

- **中文、日语等无空格语言**：这些语言天然不使用空格分隔单词，必须先通过形态分析进行分词（通常不完美）。
- **空格在语言学中的歧义性**：空格在某些语言中不代表词边界，或者空格本身是有意义的（如 Python 缩进）。
- **多语言混合文本**：编码切换（code-switching）场景中，不同语言的分词规则不同。

SentencePiece 的解决方案极其简洁：**将空格视为一个普通字符**（映射到 `_`，U+2581），直接对整个 Unicode 序列进行建模。这意味着分词器不需要任何语言特定的预处理。

### 1.4.2 Unigram 语言模型

Unigram 是 SentencePiece 支持的核心算法之一（与 BPE 并列）。它的数学基础与 BPE 完全不同——基于概率模型，而非启发式合并。

**模型定义**：Unigram 假设每个 token $$x_i$$ 的出现是独立的（因此名 unigram），则一个分词方案 $$\mathbf{x} = (x_1, \ldots, x_M)$$ 的概率为：

$$
P(\mathbf{x}) = \prod_{i=1}^{M} p(x_i), \quad \sum_{x \in V} p(x) = 1
$$

其中 $$p(x)$$ 是词表中 token $$x$$ 的**一元概率**（直接从语料统计中估计）。

对于一段文本，存在多种可能的分词方式。Unigram 选择**最大概率的分词路径**（Viterbi 解码）。设输入字符序列为 $$c_1, \ldots, c_N$$，所有可能的分词方案构成集合 $$S(c_{1:N})$$，则最优分词为：

$$
\mathbf{x}^* = \argmax_{\mathbf{x} \in S(c_{1:N})} \prod_{i=1}^{M} p(x_i)
$$

取对数后转化为最短路径问题：

$$
\mathbf{x}^* = \argmin_{\mathbf{x} \in S(c_{1:N})} \sum_{i=1}^{M} \big(-\log p(x_i)\big)
$$

这可以在 $$O(N \cdot L_{\max})$$ 时间内通过动态规划求解，其中 $$L_{\max}$$ 是词表中 token 的最大长度。

### 1.4.3 Unigram 训练：EM 算法

Unigram 的训练是一个**迭代剪枝**过程，与 BPE 的"从空到满"策略相反：

**初始化**：从一个大的种子词表开始（通常包含所有可能的子串，或通过 BPE 预训练获得）。

**迭代优化（EM 风格）**：
1. **E 步**：固定当前 token 概率 $$p(x)$$，对语料中的每条句子用 Viterbi 计算最优分词。
2. **M 步**：固定分词结果，重新估计 $$p(x)$$ 为 token $$x$$ 在最优分词中的相对频率：

$$
p^{(t+1)}(x) = \frac{\text{count}(x \text{ 在 Viterbi 分词中})}{\sum_{x' \in V} \text{count}(x' \text{ 在 Viterbi 分词中})}
$$

3. **剪枝**：计算每个 token 对总对数似然的贡献（即该 token 被使用时增加的 loss），移除贡献最小的 $$\eta \%$$ 的 token。
4. 重复 EM + 剪枝，直到词表缩小到目标大小。

形式化地，每个 token $$x$$ 的**损失贡献（loss contribution）**定义为移除该 token 后总损失的增加量：

$$
\Delta \mathcal{L}(x) = \mathcal{L}_{V \setminus \{x\}} - \mathcal{L}_V
$$

其中 $$\mathcal{L}_V = -\sum_{\text{sentences}} \log P(\text{Viterbi分词})$$。算法迭代地移除 $$\Delta \mathcal{L}(x)$$ 最小的 token，因为这些 token 对建模语料的贡献最小。

### 1.4.4 BPE vs Unigram：两种范式的对比

| 维度 | BPE | Unigram |
|------|-----|---------|
| **基本思想** | 自底向上贪心合并频次最高的相邻对 | 自顶向下概率剪枝，保留对数似然贡献最大的 token |
| **训练方向** | 从字节/字符开始逐步合并 | 从大词表开始逐步剪枝 |
| **概率模型** | 无（纯频次驱动的启发式） | 有（显式的语言模型，基于 EM 优化对数似然） |
| **编码方式** | 按合并规则顺序贪心应用 | Viterbi 动态规划，选择全局最优分词 |
| **确定性** | 确定性（相同的合并规则产生相同的编码） | 概率性（随机分词训练，但推理时用 Viterbi 确定） |
| **典型使用** | GPT 系列 | LLaMA, T5, PaLM |

SentencePiece 同时支持这两种算法。在实践中，LLaMA 使用 SentencePiece + BPE，而 T5 使用 SentencePiece + Unigram。两者在现代 LLM 中的表现差异不大，选择通常基于工程便利性和现有生态。

## 1.5 分词器的训练与推理

### 1.5.1 训练流程 (Training)

分词器的训练是一个**离线、一次性的过程**，通常发生在模型预训练之前：

```
步骤 1: 收集训练语料
        ↓
步骤 2: 文本规范化 (NFKC normalization, 小写化等)
        ↓
步骤 3: 运行分词算法 (BPE / Unigram / WordPiece)
        ↓
步骤 4: 添加特殊 token (BOS, EOS, PAD, UNK)
        ↓
步骤 5: 后处理 (如 GPT 的 <|endoftext|> token)
        ↓
步骤 6: 保存词表 + 合并规则到文件 (如 tokenizer.json, merges.txt, vocab.json)
```

**训练语料选择**：分词器的训练语料应与下游模型的预训练语料**分布一致**。如果预训练语料包含大量代码，则词表中需要包含缩进、括号对、常见变量名等。如果包含多语言文本，则需要平衡各种语言的覆盖。

**数据量的权衡**：分词器通常不需要在完整语料上训练——一个代表性样本（数 GB 文本）就足够了。实际上，HuggingFace 的 tokenizers 库默认在语料的前 5GB 上训练。更关键的是**多样性**而非**数量**。

### 1.5.2 推理流程 (Inference)

推理阶段（即实际使用分词器）包含编码和解码两个方向。

**编码（encode）**：将文本转为 token ID 序列。

```
输入文本 → 规范化 → 预分词 (可选) → 子词分割 → ID 查找
```

**预分词（Pre-tokenization）**是许多 tokenizer 的可选步骤，在应用子词算法之前将文本切分为"词"（通常按空格和标点）。SentencePiece 不使用预分词（将空格视为普通字符），而 GPT-2 BPE 使用基于正则表达式的预分词。

**解码（decode）**：将 token ID 序列转回文本。

```
token ID 序列 → 查表获取子词字符串 → 拼接 → 后处理 (如将 _ 还原为空格) → 输出文本
```

解码的复杂度远低于编码——对于 BPE，只需要查表和拼接；对于 SentencePiece，需要特殊处理 `_` 字符（还原空格）。

### 1.5.3 tokenizer.json 格式

现代 LLM 生态（尤其是 HuggingFace tokenizers）使用标准化的 `tokenizer.json` 文件来分发分词器。一个典型的 `tokenizer.json` 包含：

```json
{
  "version": "1.0",
  "truncation": null,
  "padding": null,
  "added_tokens": [
    {"id": 0, "content": "<|endoftext|>", "single_word": false, "special": true},
    {"id": 1, "content": "<|fim_prefix|>", "single_word": false, "special": true}
  ],
  "normalizer": {"type": "NFC"},
  "pre_tokenizer": {
    "type": "ByteLevel",
    "add_prefix_space": true
  },
  "model": {
    "type": "BPE",
    "vocab": {"!": 0, "\"": 1, ...},
    "merges": ["Ġ t", "Ġ a", "i n", ...]
  }
}
```

这个格式将规范化（normalizer）、预分词器（pre_tokenizer）和子词模型（model）解耦，允许灵活组合。

## 1.6 词表大小：多维度权衡

### 1.6.1 CS 336 的数据视角

从 CS 336 Lecture 3 的数据来看，现代 LLM 的词表大小有明确的规律（表中的 token 数量）：

| 单语模型 | Token 数 | 多语/生产系统 | Token 数 |
|----------|---------|-------------|---------|
| 原始 Transformer | 37,000 | mT5 | 250,000 |
| GPT | 40,257 | PaLM | 256,000 |
| GPT-2/3 | 50,257 | GPT-4 | 100,276 |
| T5/T5 v1.1 | 32,128 | Gemma 4 | 262,144 |
| LLaMA | 32,000 | Qwen 15B | 152,064 |
| | | DeepSeek | 100,000 |
| | | Yi | 64,000 |

规律一目了然：**单语词表 **30-50K** 即可，多语词表**100-250K\*\*。

### 1.6.2 小词表 vs 大词表：定量分析

设总计算预算固定。需要考虑两组相互制约的因素：

**支持小词表的理由**：
- 更小的 LM head 参数矩阵（$$|V| \times d_{\text{model}}$$），节省显存和计算。
- 每个 token 在训练中被看到的次数更多（给定相同的训练数据），embedding 学得更充分。
- 同一个词的不同形态（如 `run`/`runs`/`running`）共享底层表示，有利于泛化。

**支持大词表的理由**：
- 每个 token 代表更多信息，序列更短，同等上下文窗口覆盖更多语义。
- 自回归解码的步数更少（生成相同长度的文本需要更少的 token）。
- 对低频词的表示更精确（`photosynthesis` 不需要分解为 5 个子词）。
- 对非英语语言更友好（尤其是 CJK 字符）。

### 1.6.3 嵌入矩阵的计算成本

设模型维度为 $$d_{\text{model}}$$，词表大小为 $$|V|$$。嵌入层有两个矩阵：
- 输入嵌入 $$W_{\text{emb}} \in \mathbb{R}^{|V| \times d_{\text{model}}}$$
- 输出嵌入（LM head）$$W_{\text{out}} \in \mathbb{R}^{d_{\text{model}} \times |V|}$$

在许多模型中，这两个矩阵共享权重（weight tying），以减少参数量。无论是否共享，参数量都是 $$|V| \times d_{\text{model}}$$。以 LLaMA-7B 为例（$$d_{\text{model}} = 4096$$，$$|V| = 32000$$）：

$$
\text{参数} = 32000 \times 4096 = 1.31 \times 10^8 \approx 131\text{M}
$$

在 7B 总参数量中，嵌入矩阵占约 $$1.87\%$$——看起来不多。但如果将词表从 32K 扩展到 128K（如 LLaMA-3），嵌入参数增加到 524M，此时仅嵌入部分就占到总参数的约 $$4\%$$ 以上，这在总计算预算中是显著的。

### 1.6.4 最优词表大小：来自缩放定律的洞察

Kaplan et al. (2020) 的缩放定律实验表明，对于英文单语模型，32K-50K 的词表大小接近最优。对于多语言模型，情况更复杂：

- **深度多语言模型（如 mT5）**需要 250K 级别的词表来覆盖 100+ 种语言的字符和子词。
- **"刚好够"的多语言模型（如 GPT-4 的 100K）**通过平衡覆盖率和参数效率来适应数十种语言。
- **中文优先的模型（如 Qwen 的 152K）**需要更大的词表，因为中文字符本身的多样性远超拉丁字母。

## 1.7 特殊 Token：语义锚点

每个现代 LLM 的分词器都定义了一组特殊 token，它们不在常规文本中出现，而是承载特定的语义或控制功能。

### 1.7.1 核心特殊 Token

| Token | 含义 | 位置 | 用途 |
|-------|------|------|------|
| `[BOS]` / `<s>` | Beginning of Sequence | 序列开头 | 标记句子起始；在某些架构中作为生成起始信号 |
| `[EOS]` / `</s>` | End of Sequence | 序列末尾 | 标记句子结束；在自回归生成中，产生此 token 表示终止 |
| `[PAD]` | Padding | 填充位置 | 将不等长序列填充到相同长度；其计算应被忽略（通过 attention mask） |
| `[UNK]` | Unknown | 替代未登录词 | 传统做法：将词表外的 token 映射为 `[UNK]`。现代字节级 tokenizer 不存在此问题 |
| `[SEP]` | Separator | 句子之间 | BERT 使用此 token 分隔两个句子（NSP 任务） |
| `[MASK]` | Mask | 被遮蔽位置 | BERT 的 MLM 训练——模型预测此位置的原始 token |
| `[CLS]` | Classification | 序列开头 | BERT 使用此 token 的隐藏表示作为整句分类的特征 |

### 1.7.2 现代对话模型的特殊 Token

随着 LLM 从"预测下一个 token"进化为对话智能体，特殊 token 的角色变得更为复杂。以 LLaMA-3 的 chat template 为例：

```
<|begin_of_text|><|start_header_id|>system<|end_header_id|>

You are a helpful assistant.<|eot_id|><|start_header_id|>user<|end_header_id|>

What is tokenization?<|eot_id|><|start_header_id|>assistant<|end_header_id|>
```

这些特殊 token（`<|begin_of_text|>`、`<|start_header_id|>`、`<|eot_id|>` 等）本质上是**对话结构的 markup**。它们告诉模型：
1. 当前在哪一轮对话
2. 谁在说话（system / user / assistant）
3. 一段话什么时候结束

这些 token 的嵌入向量在预训练后阶段（如监督微调 SFT 或 RLHF）中学习，与普通 token 共享同一个嵌入矩阵。

## 1.8 多语言分词：陷阱与实践

### 1.8.1 不同语言的分词效率差异

分词效率可以用**每 token 平均字符数**来量化。对于英语，BPE 分词器通常能达到 3.5-4.5 字符/token。但对于其他语言：

| 语言 | 每 token 字符数（典型） | 说明 |
|------|------------------------|------|
| 英语 | 3.5-4.5 | BPE 分词对英语效果很好 |
| 法语/西班牙语 | 3.0-4.0 | 接近英语，但变位形式更多 |
| 德语 | 2.5-3.5 | 复合词（如 "Rechtsschutzversicherungsgesellschaften"）导致碎片化 |
| 中文 | 0.6-1.5 | 每个汉字可能成为一个独立的 token |
| 日语 | 0.8-1.5 | 同中文，但假名连写稍微提高效率 |
| 韩语 | 0.7-1.3 | 谚文字母组合可能被拆分 |
| 阿拉伯语 | 2.5-3.5 | 连写形式和丰富的变位导致碎片化 |

中文的分词效率约为英文的 1/3——这意味着，在相同的 4096 token 上下文窗口下，中文用户能放入的有效信息量只有英文用户的约 1/3。这是 LLM 在多语言平等性方面的一个根本性结构问题。

### 1.8.2 数字处理：一个普遍性难题

数字的 tokenization 是一个微妙但重要的问题。考虑以下数字的 BPE 分词（GPT-2 tokenizer）：

```
"1234"    → ["123", "4"]
"12345"   → ["123", "45"]
"123456"  → ["12", "34", "56"]
"3.14159" → ["3", ".", "14", "15", "9"]
```

数字没有固定的"正确"分词方式——因为数字空间中不存在自然的分割边界。这带来了两个问题：

1. **算术能力受损**：模型需要学习在 token 空间中"对齐"数字的位值。`123 + 456` 可能被 tokenized 为 `[12, 3, +, 45, 6]`，这使得模型难以直接在 token 级别执行进位操作。
2. **数值比较困难**：模型无法通过简单的词表查看来判断 `"1234"` 和 `"1000"` 的大小关系——它必须通过 token 的上下文表征来隐式学习数值大小。

**改进方向**：最近的模型（如 PaLM、Gemma）倾向于对数字强制使用**逐位分词**（digit-by-digit tokenization）——将每个数字 0-9 作为独立的 token，以此保证模型可以精确地操作每一位数字。这在推理基准测试（如 GSM8K）中带来了显著的性能提升。

### 1.8.3 代码分词

代码的分词面临独特的挑战：

- **缩进语义**：Python 代码的缩进具有语义（决定作用域），但被 BPE 视为普通空白字符。
- **标识符碎片化**：`generateResponseForUser` 可能被分成 `["generate", "Response", "For", "User"]`，使得模型难以学习变量名的确切组成。
- **编程语言特定语法**：`!=`、`<=`、`->` 等运算符如果被拆分为多个 token，会破坏其原子语义。

现代代码 LLM（如 Code LLaMA、StarCoder）采用以下策略缓解这些问题：
- 在训练语料中包含充足的代码，让 BPE 自然学习到常见的代码 token（如 `def `、`self.`、`import `）
- 使用更大的词表（100K+）来容纳更多完整的标识符
- 在预分词阶段对代码特殊字符（如 `.`、`->`）进行保护

### 1.8.4 空白字符处理

空白字符（whitespace）在不同 tokenizer 中的处理方式是一个重要但容易被忽视的实现细节：

**BPE 的预处理**：GPT-2 的 BPE tokenizer 使用正则表达式 `'s|'t|'re|'ve|'m|'ll|'d| ?\p{L}+| ?\p{N}+| ?[^\s\p{L}\p{N}]+|\s+(?!\S)|\s+` 进行预分词。这条正则的含义是：
- 分离常见的英文缩写（`'s`, `'t` 等）
- 将连续的字母序列作为一个预分词单元（前面可能有一个空格）
- 将连续的数字序列作为一个单元
- 将连续的标点作为一个单元
- 单独处理末尾的空白字符

注意：空格被**合并到后续的 token 开头**（除非是末尾空白）。这意味着 token `" hello"`（前面有空格）和 token `"hello"`（没有空格）在词表中是**不同**的。这种设计减少了词表空间中的重复——词表不需要同时存储 `hello` 和 ` hello`。

**SentencePiece 的解决方案**：将所有空格替换为元字符 `_`（U+2581, LOWER ONE EIGHTH BLOCK），然后将整个文本作为连续的字符流处理。解码时，只需将 `_` 还原为空格。这种方法无需预分词，适用于所有语言。

## 1.9 本章小结

分词是语言模型的第一道工序，也是连接"人类语言"和"机器表示"的桥梁。本章覆盖了以下核心内容：

1. **BPE（Byte-Pair Encoding）**：自底向上的贪心合并算法，通过统计相邻符号对的频次来构建子词词表。合并规则构成一棵二叉树，天然支持对未登录词的分解。
2. **Unigram 语言模型**：自顶向下的概率剪枝算法，基于 EM 优化保留对数似然贡献最大的 token。推理时使用 Viterbi 动态规划选择全局最优分词。
3. **训练与推理分离**：分词器训练是离线的一次性过程，需要在语料多样性和计算成本之间权衡。推理时编码按合并规则顺序贪心（BPE）或 Viterbi 解码（Unigram）。
4. **词表大小的多维度权衡**：单语模型 30-50K，多语模型 100-250K。更大的词表意味着更短的序列（但更多的嵌入参数），这一 trade-off 需要通过实验确定。
5. **特殊 token 的语义角色**：从传统的 `[BOS]`/`[EOS]`/`[PAD]` 到现代对话模板的角色标记，特殊 token 是 LLM 理解"结构"的锚点。
6. **多语言公平性**：中文的分词效率约为英语的 1/3，这是一个系统工程层面的不公平性——相同的 token 预算（和 API 计费）下，不同语言的用户获得的有效信息量不同。

在下一章中，我们将探讨如何在分词的基础上构建语言模型——从自回归分解到困惑度评估。
