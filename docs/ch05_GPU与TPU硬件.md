## 5.1 本章概要

**核心问题**: 现代大语言模型的训练与推理依赖海量算力。从 GPT-3 的 1750 亿参数到 DeepSeek-V3 的万亿级 MoE 架构, 模型规模每 18 个月翻 10 倍。传统的 CPU 计算范式(Dennard 缩放)早已触顶, GPU 的并行缩放在过去十年驱动了超过 1000 倍的计算增长。本章要回答的问题是: **GPU 和 TPU 究竟是如何工作的? 为什么矩阵乘法在它们上面如此之快? 以及如何从硬件倒推算法设计?**

**动机**: 不理解硬件就无法写出高效的 ML 代码。一个看似简单的操作——比如注意力机制——在 GPU 上的朴素实现与硬件感知实现之间可能有 **5-10 倍的性能差距**。FlashAttention 就是最好的例子: 通过理解 GPU 的内存层次和计算特性, Dao 等人将注意力机制的计算从显存受限转变为计算受限, 实现了数量级的加速。本章将为读者建立从硅片到算法的完整直觉。

**本章主线**: 本章分为三个部分:
1. **GPU 深度剖析**(5.2-5.4 节): 从 GPU 与 CPU 的本质差异出发, 自底向上解析 GPU 的硬件架构——流式多处理器(SM)、Warp 调度、内存层次(HBM、L2、L1、共享内存、寄存器), 以及 Tensor Core 的矩阵乘法加速机制与低精度计算革命。
2. **性能分析框架**(5.5-5.6 节): 引入 Roofline 模型和算术强度概念, 建立判断计算受限 vs 访存受限的严格分析框架; 对比 TPU 的脉动阵列架构与 GPU 的 SIMT 模型, 并介绍 CUDA 与 Triton 编程模型。
3. **优化技术与案例分析**(5.7-5.8 节): 系统梳理六大 GPU 优化技术——控制流分歧、低精度计算、算子融合、重计算、内存合并、分块(Tiling); 最后以 FlashAttention 为案例, 展示如何将分块、融合和在线 softmax 等多个技术组合为一个硬件感知的高效算法。

读完本章, 你应该能够回答: 为什么 A100 有 108 个 SM, 每个 SM 有 64 个 Warp 槽位, 这种数量关系是如何影响性能的? 为什么 FP16 矩阵乘法比 FP32 快 10 倍以上? 为什么 TPU 没有 Warp 概念? 以及 FlashAttention 究竟"快"在哪里?

## 5.2 GPU: 大规模并行的计算引擎

### 5.2.1 计算范式的分野: GPU vs CPU

CPU 和 GPU 代表了两种截然不同的设计哲学:

| 维度 | CPU | GPU |
|:---|:---|:---|
| 核心数量 | 少(几个到几十个) | 极多(数千到上万个 ALU) |
| 单线程性能 | 极高(深度流水线、分支预测、乱序执行) | 较低(顺序执行, 简化控制逻辑) |
| 优化目标 | 延迟(Latency) | 吞吐量(Throughput) |
| 分支支持 | 强(大容量缓存、分支预测器) | 弱(控制逻辑占用面积最小化) |
| 缓存层次 | 大容量 L1/L2/L3 | 小容量、高度并行化的共享内存 |
| 线程切换 | 重(上下文切换开销大) | 轻(零开销 Warp 切换) |

CPU 为**少量快速线程**优化——每个线程应尽可能快地完成。GPU 为**海量线程**优化——在任何给定时刻, GPU 上可能有数万个线程处于飞行状态 (in-flight), 当一个 Warp 因访存而停顿时, GPU 调度器可以在一个时钟周期内切换到另一个 Warp, 零开销地隐藏访存延迟。

这种设计差异从根本上决定了两种处理器的适用场景。考虑矩阵乘法 $$C = A \times B$$, 其中 $$A \in \mathbb{R}^{M \times K}$$, $$B \in \mathbb{R}^{K \times N}$$。输出矩阵 $$C$$ 的每个元素都是 $$K$$ 次乘加运算的结果。这些 $$M \times N$$ 个输出元素之间**没有任何数据依赖**——它们可以完全并行地计算。这正是 GPU 的"甜区": 将每个输出元素分配给一个线程, 数千个线程同时执行相同的乘加指令(只是数据不同), 即 **SIMT(Single Instruction, Multiple Threads)模型**。

### 5.2.2 GPU 的解剖: SM 与 SP

现代 NVIDIA GPU(如 A100、H100)的顶层架构如下:

```
GPU
├── GPC (Graphics Processing Cluster) × N
│   ├── TPC (Texture Processing Cluster) × M
│   │   └── SM (Streaming Multiprocessor) × 2
│   └── Raster Engine
├── L2 Cache (共享, 所有 SM 可访问)
├── HBM (High Bandwidth Memory, 片外显存)
└── NVLink / PCIe 接口
```

**SM(Streaming Multiprocessor, 流式多处理器)**是 GPU 的核心计算单元。以 A100 为例:
- 每个 A100 有 **108 个 SM**(实际是 108 个, 部分被屏蔽用于良率)
- 每个 SM 包含 **64 个 CUDA Core**(FP32/INT32 ALU) + **4 个 Tensor Core**(第三代)
- 每个 SM 有独立的 **L1 缓存**(192 KB, 可配置为共享内存)和**寄存器文件**(65536 个 32 位寄存器)
- 每个 SM 有独立的 Warp 调度器, 可同时管理 64 个 Warp(2048 个线程)

**SP(Streaming Processor, 流式处理器)** 即单个 ALU。一个 CUDA Core 就是一个 SP。在现代 GPU 中, SP 被组织为更粗粒度的 SM——关键架构洞察是: **多个 SP 共享同一个指令获取单元、同一个 Warp 调度器和同一块共享内存**。这种分组设计使得 GPU 可以用较少的控制逻辑驱动大量 ALU, 从而将更多的芯片面积留给计算单元和存储单元。

**关键数值直觉(A100)**:
- 总 CUDA Core: $$108 \times 64 = 6912$$
- 总 Tensor Core: $$108 \times 4 = 432$$
- 总线程容量(每 SM 2048 线程): $$108 \times 2048 = 221184$$ 个并发线程
- FP16 Tensor Core 峰值: 312 TFLOPs/s
- HBM2e 带宽: 2039 GB/s

### 5.2.3 GPU 执行模型: 线程、线程块与 Warp

GPU 的执行模型涉及三个层次:

**线程(Thread)**:
- 执行工作的最小单元
- 每个线程执行相同的内核(Kernel)代码, 但处理不同的数据(SIMT)
- 每个线程拥有**私有寄存器**(通常最多 255 个 32 位寄存器)
- 线程是"轻量级"的——创建、销毁和切换的开销极低

**线程块(Thread Block)**:
- 一组协作的线程(三维: 最大 1024 个线程/块)
- 一个线程块**必须**在一个 SM 上执行(不能跨 SM)
- 同一线程块内的所有线程共享**共享内存**(Shared Memory)
- 线程块内的线程可以通过 `__syncthreads()` 进行同步
- 线程块之间**不能**直接通信——所有跨块的数据交换必须通过全局内存(慢!)

**Warp(线程束)**:
- 线程以 Warp 为单位调度执行: **32 个连续编号的线程组成一个 Warp**
- 同一个 Warp 中的所有线程在**同一时钟周期**执行**同一条指令**(SIMT 的本质)
- Warp 是 GPU 调度的基本单位——Warp 调度器在每个时钟周期选择一个就绪的 Warp 发射指令
- 如果 Warp 中的线程执行不同的分支路径, 两条路径会被**串行化执行**(控制流分歧)

**Warp 调度的核心机制**:

每个 SM 中有多个 Warp 调度器。以 A100 为例, 每个 SM 有 4 个 Warp 调度器, 每个调度器每个时钟周期可发射一条指令到其管理的 Warp 组。关键之处在于: SM 中驻留的 Warp 数量远大于调度器数量——这是刻意为之的。当一个 Warp 因等待内存访问而停顿时(可能需要数百个时钟周期), 调度器立即切换到另一个就绪的 Warp。这种**零开销 Warp 切换**是 GPU 隐藏内存延迟的核心机制。

**占用率(Occupancy)**定义为:
$$
\text{Occupancy} = \frac{\text{SM 上活跃 Warp 数}}{\text{SM 最大 Warp 容量}}
$$

高占用率意味着调度器有更多的 Warp 可供选择, 从而更有效地隐藏访存延迟。但占用率并不是越高越好——过高的占用率会减少每个线程可用的寄存器数量(导致寄存器溢出到 L1/共享内存), 反而降低性能。

### 5.2.4 GPU 内存模型: 从线程到全局

GPU 的内存模型是多层次的, 每层对应不同的可见性范围:

| 内存类型 | 可见范围 | 大小(每 SM 典型值) | 延迟 | 带宽 |
|:---|:---|:---|:---|:---|
| 寄存器(Register) | 单线程 | 每个 SM 65536 × 32-bit | ~0 周期 | 极高 |
| 共享内存(Shared Memory) | 单线程块 | 可达 164 KB/ SM | ~20-30 周期 | ~10 TB/s |
| L1 缓存 | 单 SM | 与共享内存共享 256 KB | ~30-50 周期 | ~5 TB/s |
| L2 缓存 | 全 GPU | 40 MB (A100) | ~200-300 周期 | ~4 TB/s |
| 全局内存(HBM/DRAM) | 全 GPU + CPU | 40/80 GB (A100/H100) | ~600-800 周期 | ~2-3 TB/s |

**内存层次的核心权衡**: SRAM(共享内存/L1 缓存)比 DRAM(HBM 全局内存)快约 8 倍, 但单位容量的成本高约 100 倍。这也是为什么 GPU 的片上 SRAM 容量远小于片外 HBM。

**关键洞察**: 数据越靠近 SM, 访问越快。L1 缓存和共享内存在物理上位于 SM 内部; L2 缓存位于芯片上所有 SM 之间(On-Die); 全局内存(HBM 堆栈)是独立的芯片, 通过中介层(Interposer)与 GPU 芯片相连。这种物理距离直接映射为访问延迟的层级差。

**寄存器的特殊性**: 寄存器是唯一对每个线程私有的存储。每个线程的寄存器对其他线程完全不可见——即使是同一 Warp 内的相邻线程。这意味着线程间的数据交换必须通过共享内存或全局内存进行。

## 5.3 GPU 内存层次: 从 HBM 到寄存器的完整图谱

### 5.3.1 数字的威力: 带宽演进

让我们通过具体的硬件数字来建立对 GPU 内存系统的直觉。

**A100 (2020, Ampere, 7 nm)**:
- HBM2e: 80 GB, 2039 GB/s
- L2 Cache: 40 MB
- L1/Shared Memory: 192 KB/SM × 108 SM ≈ 20.3 MB (可配置)
- Register File: 65536 × 32-bit/SM × 108 SM = 27 MB
- FP16 Tensor Core Peak: 312 TFLOPs/s

**H100 (2022, Hopper, 4 nm)**:
- HBM3: 80 GB, 3350 GB/s
- L2 Cache: 50 MB
- L1/Shared Memory: 256 KB/SM × 132 SM = 33 MB
- Register File: 65536 × 32-bit/SM × 132 SM ≈ 34 MB
- FP16 Tensor Core Peak: 989.5 TFLOPs/s

**H200 (2024, Hopper Refresh)**:
- HBM3e: 141 GB, 4800 GB/s
- 其他参数与 H100 相同

**B200 (2024, Blackwell, 4 nm)**:
- HBM3e: 192 GB, 8000 GB/s
- FP8 Tensor Core Peak: 4.5 PFLOPs/s
- L2 Cache: 大幅增加(确切值未公开)

**关键趋势**: FLOPs 的增速远快于内存带宽。从 A100 到 H100, FP16 算力增长了约 3.2 倍(312 → 989.5 TFLOPs/s), 而内存带宽仅增长约 1.6 倍(2039 → 3350 GB/s)。这意味着**将数据喂给计算单元的难度越来越大**——这就是著名的"内存墙"(Memory Wall)问题。

### 5.3.2 SRAM vs DRAM: 物理根源

为什么片上 SRAM 比片外 DRAM 快 8 倍? 原因在物理层面:

- **SRAM**(静态随机存取存储器): 由 6 个晶体管构成一个存储单元, 速度快、功耗低, 但占用面积大。制造在逻辑工艺上, 与计算单元在同一步骤生产。
- **DRAM**(动态随机存取存储器): 由 1 个晶体管 + 1 个电容构成, 密度高、成本低, 但需要周期性地刷新(电容漏电)。DRAM 使用独立的优化工艺, 必须通过中介层或 PCB 走线与 GPU 芯片连接——物理距离和多级互联导致高延迟和有限带宽。

**HBM(High Bandwidth Memory, 高带宽内存)**是 DRAM 的一种特殊封装形式: 多层 DRAM 芯片垂直堆叠, 通过硅通孔(TSV, Through-Silicon Via)互联, 再通过硅中介层与 GPU 芯片连接。相比传统的 GDDR(通过 PCB 走线), HBM 的总线宽度可达 1024 位(A100 有 5 个 HBM 堆栈, 每堆栈 1024 位), 从而实现数 TB/s 的带宽。

**关键架构决策**: GPU 设计者必须在有限的芯片面积上决定多少给 SRAM(共享内存/L1/L2)、多少给 ALU、多少给 Tensor Core。NVIDIA 的选择是: 给计算(Tensor Core + CUDA Core)分配更多面积, 而将数据供应的责任部分推给软件——即通过 Tiling 等技术让程序员显式地管理数据在内存层次中的移动。

### 5.3.3 共享内存: 程序员可控的片上 SRAM

共享内存(Shared Memory)是 GPU 编程中最重要的一块内存空间:

- **物理位置**: 在 SM 内部, 紧邻 ALU
- **逻辑所有者**: 一个线程块(同一块内的所有线程均可访问)
- **速度**: 约 10 TB/s 的片上带宽(相比 HBM 的 2-3 TB/s)
- **大小限制**: 每个 SM 约 164-228 KB(取决于 GPU 代际和配置), 共享给同一 SM 上的多个线程块
- **编程模型**: 程序员用 `__shared__` 关键字声明, 显式控制数据的加载和存储

共享内存是 Tiling 优化的核心——程序员将全局内存中的数据加载到共享内存, 在共享内存中重用这些数据, 最后将结果写回全局内存(详见 5.7 节)。

**Bank Conflict(存储体冲突)**是使用共享内存时必须注意的问题。共享内存被组织为 32 个 Bank, 每个 Bank 每个周期可服务一个地址。如果 Warp 中的 32 个线程访问的地址映射到同一个 Bank, 这些访问会被串行化——称为 Bank Conflict。以 32 路冲突为例(最坏情况), 需要 32 个周期来完成本应 1 个周期的访问——性能下降 32 倍。

## 5.4 Tensor Core 与低精度计算革命

### 5.4.1 从可编程着色器到专用矩阵乘法器

NVIDIA GPU 的起源是图形渲染——可编程着色器(Programmable Shaders)。早期的研究者发现着色器可以用来做通用计算(GPGPU), 包括矩阵乘法。2006 年 CUDA 的发布将这一非正规实践规范化为一个通用计算平台。

但真正的转折点出现在 **Tensor Core** 的引入。Tensor Core 是专门为矩阵乘法-累加(Matrix Multiply-Accumulate)设计的硬件电路, 首次出现在 Volta 架构(V100, 2017 年):

$$
D = A \times B + C
$$

其中 $$A, B$$ 是输入矩阵, $$C, D$$ 是累加器。在一个 GPU 时钟周期内, 一个 Tensor Core 可以执行 $$4 \times 4 \times 4 = 64$$ 次乘加运算(即 128 FLOPs)。多个 Tensor Core 并行工作, 实现了 **10 倍于普通浮点运算的矩阵乘法吞吐量**。

**Tensor Core 的演进**:

| 架构 | 代际 | Tensor Core 精度 | 关键特性 |
|:---|:---|:---|:---|
| Volta (V100) | 第一代 | FP16 → FP32 accumulate | 首次引入 Tensor Core |
| Turing (T2000) | 第二代 | INT8, INT4, FP16 | 推理加速 |
| Ampere (A100) | 第三代 | TF32, BF16, FP16, INT8 | TF32: 19-bit mantissa for DL |
| Hopper (H100) | 第四代 | FP8 (E4M3/E5M2) | 吞吐量翻倍 vs A100 |
| Blackwell (B200) | 第五代 | MXFP8 (E4M3), MXFP4 | 微缩放格式, 更高密度 |

### 5.4.2 精度模式全景

现代 GPU 支持多种数值精度模式, 每种都在"精度 vs 速度 vs 内存"之间做了不同的权衡:

**FP32 (Float32, 32 位单精度浮点)**:
- 格式: 1 符号位 + 8 指数位 + 23 尾数位
- 范围: $$\sim 10^{-38} \sim 10^{38}$$
- 每个数占 4 字节
- 基准精度, 训练中用于存储主权重副本(Master Weights)

**TF32 (TensorFloat-32, A100 引入)**:
- 格式: 1 符号位 + 8 指数位 + 10 尾数位 = 19 位有效
- 每个数占 4 字节(存储)但内部精度为 19 位
- 与 FP32 相同的指数范围, 但尾数精度降低
- 无需修改代码即可获得 A100 上 8 倍的矩阵乘法加速(相比 V100 FP32)

**FP16 (Float16, 16 位半精度)**:
- 格式: 1 符号位 + 5 指数位 + 10 尾数位
- 范围: $$\sim 10^{-8} \sim 65504$$
- 每个数占 2 字节
- 范围有限, 训练中容易出现上溢/下溢, 需要损失缩放(Loss Scaling)

**BF16 (Brain Float16, A100 支持)**:
- 格式: 1 符号位 + 8 指数位 + 7 尾数位
- 范围: 与 FP32 相同($$\sim 10^{-38} \sim 10^{38}$$)
- 每个数占 2 字节
- 范围大但尾数精度低: 解决了 FP16 的上溢/下溢问题, 是当前 LLM 训练的主流格式

**FP8 (Float8, H100 引入)**:
- 两种变体:
  - E4M3: 4 指数位 + 3 尾数位 — 更高精度, 用于前向传播
  - E5M2: 5 指数位 + 2 尾数位 — 更大范围, 用于反向传播梯度
- 每个数仅占 1 字节
- H100 的 FP8 Tensor Core 吞吐量是 FP16 的 2 倍

**MXFP8 (Microscaling FP8, Blackwell 引入)**:
- 格式: E4M3(更多尾数位) + 每 32 个元素共享一个 FP8(E8M0)缩放因子
- 核心创新: 不再是每个元素各自表示, 而是一组元素共享缩放因子
- 缩放因子本身也是 FP8(E8M0 格式: 仅指数, 无符号, 无尾数)
- 后果: 转置(Transpose)操作变得复杂——缩放因子需要跟随数据移动
- 实践中有趣的现象: 并非所有权重都使用 MXFP8; 转置操作需要单独量化

**MXFP4 (Microscaling FP4, Blackwell 支持)**:
- 格式: 每 16 个元素共享一个 E4M3 缩放因子
- 每个数据值仅 4 位——总共只能表示 16 个值!
- 但配合逐块的缩放因子, 可以在 LLM 推理中以极低精度运行部分层
- 这是精度的极致压缩: 可表示的值域极小, 但数据的分布信息由缩放因子捕捉

### 5.4.3 低精度为何有效: 算术强度视角

低精度计算的收益是双重的:

**(1) 减少内存占用**:
同样的模型参数, FP16 比 FP32 少用一半显存。对于 175B 参数的 GPT-3: FP32 需 700 GB, FP16 仅需 350 GB——这是 H100(80 GB)装下模型的边缘条件。

**(2) 提高算术强度**:
考虑一个逐元素 ReLU 操作 $$x = \max(0, x)$$ 对一个长度为 $$n$$ 的向量:

**FP32 情况**:
- 内存访问: 1 读(x) + 1 写(当 x < 0 时), FP32 = 4 字节/操作
- 计算: 1 次比较 + 1 FLOP
- 算术强度: 8 字节/FLOP ← 极低

**FP16 情况**:
- 内存访问: 1 读(x) + 1 写(当 x < 0 时), FP16 = 2 字节/操作
- 计算: 1 次比较 + 1 FLOP
- 算术强度: 4 字节/FLOP ← 减半, 但依然很低

对于逐元素操作(逐元素加法、ReLU、GELU 等), 算术强度极低——这些操作几乎总是访存受限的。低精度虽然将访存量减半, 但算术强度仍然远离计算受限区域。这也是为什么**算子融合**(Fusion)对逐元素操作如此重要——通过将多个逐元素操作合并为一个内核调用, 可以减少显存往返次数(见 5.7 节)。

但对于矩阵乘法, 情况截然不同。以 $$C = A \times B$$ 为例:

**FP16 矩阵乘法**:
- 内存访问: $$2(MK + KN + MN)$$ 字节
- 计算: $$2MKN$$ FLOPs (1 Multiply + 1 Add = 2 FLOPs)
- 算术强度: $$\frac{2MKN}{2(MK + KN + MN)} \approx \min(M, N)$$ 当 $$K$$ 较大时
- 对于 $$M=N=K=4096$$: AI ≈ 1365 FLOPs/Byte

在 H100 上, Ridge Point = 295 FLOPs/Byte, 因此矩阵乘法的算术强度远超临界值——它是**计算受限**的。而 Tensor Core 这些专用硬件就是为了在这类计算受限的场景中榨取每一分 FLOPs。

## 5.5 TPU: 脉动阵列与系统级协同设计

### 5.5.1 高层对比: GPU vs TPU

从足够高的抽象层看, GPU 和 TPU 是相似的: 都有轻量级的控制逻辑、高速的大矩阵乘法单元和快速的内存。但两者的设计哲学有根本差异:

| 维度 | GPU (NVIDIA H100) | TPU (Google v5p) |
|:---|:---|:---|
| 计算架构 | SIMT: 多 SM 独立调度 | 脉动阵列(Systolic Array): 单一大型 MXU |
| 基本计算单元 | SM × 132, 每 SM 含 4 个 Tensor Core | 一个大型 MXU(矩阵乘法单元) |
| 线程模型 | Warp(32 线程), 线程块, 网格(Grid) | 仅线程块(无 Warp) |
| 内存层次 | HBM → L2 → L1 → Shared → Reg | HBM → 片上缓冲区 → 脉动阵列 |
| 编程模型 | CUDA(C++) / Triton(Python) | JAX / TensorFlow(XLA 编译器) |
| 网络互联 | NVLink / NVSwitch (单机) + InfiniBand (跨机) | ICI (Inter-Chip Interconnect, 片间互联) |
| 单芯片 MatMul 性能 | 相近量级 | 相近量级 |
| 非 MatMul 操作性能 | 灵活, 各 SM 可独立执行不同 Kernel | 不够灵活, 非 MatMul 操作可能成为瓶颈 |

**核心差异**: GPU 有更多但更小的矩阵乘法单元(每个 SM 的 Tensor Core), 而 TPU 有一个更大但数量更少的矩阵乘法单元(MXU)。两者的 MatMul 总吞吐量相近, 但**没有 Warp 的概念**是 TPU 最大的简化——这带来了 MatMul 与非 MatMul 操作之间的编程模型权衡。

### 5.5.2 TPU 的脉动阵列架构

TPU 的核心是 **MXU(Matrix Multiply Unit, 矩阵乘法单元)**——一个脉动阵列(Systolic Array)。脉动阵列的工作原理如下:

想象一个 $$128 \times 128$$ 的处理单元(PE)网格。数据不是一次性加载到所有 PE 中, 而是以**脉动**(systolic)的方式流过阵列:

1. 权重矩阵的元素从顶部**预加载**到 PE 阵列中(每个 PE 存储一个权重值)
2. 输入激活从左边界泵入, 在每个时钟周期**向右传递**到相邻 PE
3. 每个 PE 在每个周期执行一次乘加: $$accum = accum + input \times weight$$
4. 部分和向下传递, 最终从底部泵出

这种脉动数据流有几个关键优势:
- **数据重用最大化**: 权重驻留在 PE 中, 被反复使用——不必反复从内存加载
- **互联简化**: 每个 PE 只与紧邻通信(左、右、上、下), 无需全局互联网络
- **高时钟频率**: 简化的互联允许更高的时钟频率——因为没有长距离走线的延迟限制
- **确定性的性能**: 脉动阵列的性能完全由阵列大小和时钟频率决定, 可精确预测

TPU 的缺点也很明显: **灵活性不足**。非矩阵乘法操作(如逐元素操作、归约、排序等)在 TPU 上要么无法高效执行, 要么需要通过编译器(XLA)进行复杂的模式匹配和替换。这也是为什么 TPU 通常搭配 JAX 或 TensorFlow 使用——XLA 编译器可以将高级操作编译为针对脉动阵列优化的低级指令序列。

### 5.5.3 GPU 和 TPU 的深层对比

**MatMul 场景**(两者表现相近):
100 TFLOPS 级别的矩阵乘法吞吐量。对于 Transformer 中占主导地位的矩阵乘法(Attention 的 QKV 投影、MLP 的前馈网络), GPU 和 TPU 的硬件效率都在 70-80% 的峰值利用率范围内。

**非 MatMul 场景**(GPU 通常优于 TPU):
- GPU 的每个 SM 可以独立执行不同的内核——例如, 一些 SM 在做 MatMul, 另一些在做 LayerNorm
- TPU 的单一大型 MXU 对非 MatMul 操作没有加速优势, 这些操作可能成为系统瓶颈
- GPU 上的 Warp 抽象提供了更细粒度的调度(32 线程), 而 TPU 在块级别调度

**系统级差异**:
- GPU 的 NVLink + InfiniBand 互联是通用的——支持任意拓扑、任意通信模式
- TPU 的 ICI 是定制化的——专门为 TPU 间的数据交换优化, 但在混合环境中不够灵活
- 这方面将在后面的并行计算章节中详细讨论

### 5.5.4 GPU 模型的优势总结

回看 GPU 的三大优势:

1. **通过增加 SM 轻松扩展**: 更多 SM 意味着更高的并行度, 扩展路径清晰——从 A100(108 SM)到 H100(132 SM)再到 B200(更多 SM)。工作负载可以在几乎不修改代码的情况下获得加速。

2. **SIMT 模型的编程便利性**: 程序员可以编写标量代码, GPU 硬件自动将相同指令广播到 Warp 中的所有线程。在 CUDA 中, 你写 C++ 代码, 就像每个线程在独立运行一样——这种抽象大大降低了 GPU 编程的门槛。

3. **轻量级线程**: GPU 线程的创建、销毁和切换几乎零开销。这允许 GPU 启动成百上千个线程块, 每个线程块包含数百个线程——总共数十万个飞行中的线程。当一个线程因访存而等待时, 调度器立即切换——在宏观上, 计算单元始终保持忙碌。

## 5.6 Roofline 模型: 从硬件到性能极限

### 5.6.1 算术强度: 计算量 / 访存量

**算术强度(Arithmetic Intensity, AI)**是 Roofline 分析的核心概念, 定义为:

$$
\text{AI} = \frac{\text{FLOPs}}{\text{Bytes Moved}}
$$

其中 "Bytes Moved" 指从主存(HBM)读取和写入的数据总量。算术强度的单位是 FLOPs/Byte——它衡量的是**每个内存字节上的平均计算量**。

算术强度高意味着"有很多计算可以做, 但数据很少"——这是计算受限的场景。算术强度低意味着"计算很少, 但需要搬很多数据"——这是访存受限的场景。

**关键认知**: 不同的算子有不同的算术强度, 而且差距可能在 3 个数量级以上:

| 算子 | 近似 AI (FLOPs/Byte) | 瓶颈类型 |
|:---|:---|:---|
| 逐元素操作 (ReLU, Add, GELU) | 0.25-1 | 强 Memory-Bound |
| LayerNorm / RMSNorm | 1-5 | Memory-Bound |
| GEMV (batch_size=1, 如单 token Decode) | 1-10 | Memory-Bound |
| Self-Attention (小序列) | 10-50 | 过渡区域 |
| GEMM (矩阵乘法, 中等规模) | 100-500 | Compute-Bound |
| GEMM (矩阵乘法, 大规模) | 1000+ | 强 Compute-Bound |

### 5.6.2 Roofline 模型的定义与构造

Roofline 模型将硬件的能力表示为一个两段函数:

1. **访存受限区(Memory-Bound, 上升斜坡)**:
   $$
   \text{Attainable Performance} = \text{AI} \times \text{Peak Memory Bandwidth}
   $$
   在这个区域, 算力没有被充分利用——性能受内存带宽限制, 提升 AI 可以线性提升性能。

2. **计算受限区(Compute-Bound, 水平平台)**:
   $$
   \text{Attainable Performance} = \text{Peak Compute Throughput}
   $$
   在这个区域, 内存带宽充足——性能受计算单元峰值限制, 唯一的出路是减少计算量或使用更快的计算硬件。

两者的分界点是 **Ridge Point**(临界算术强度):
$$
\text{AI}_{\text{ridge}} = \frac{\text{Peak Compute Throughput}}{\text{Peak Memory Bandwidth}}
$$

**H100 的 Roofline 实例**:

$$
\text{AI}_{\text{ridge}} = \frac{989.5 \times 10^{12} \text{ FLOPs/s}}{3.35 \times 10^{12} \text{ Bytes/s}} \approx 295 \text{ FLOPs/Byte}
$$

(注意: H100 的 bf16 理论峰值为 1979 TFLOPs/s, 但实际可用约 50%, 即 989.5 TFLOPs/s。不同精度、稀疏稠密度、以及矩阵形状会影响实际可达峰值。)

这意味着: 一个算子的算术强度如果低于 295, 它就跑不满算力(访存受限); 如果高于 295, 它才能充分榨取 GPU 的浮点能力(计算受限)。

### 5.6.3 深度推理: 为什么 Decode 是 Memory-Bound 而 Prefill 是 Compute-Bound

应用 Roofline 模型到 LLM 推理的两个阶段, 可以精确推导出它们的性能瓶颈。考虑 MLP 层中的矩阵乘法 $$O = I \times W$$, 其中 $$I \in \mathbb{R}^{B \times D}$$, $$W \in \mathbb{R}^{D \times F}$$:

- 计算量: $$2BDF$$ FLOPs
- 访存量: $$2(BD + DF + BF)$$ 字节

当 $$B \ll D, F$$ 时(权重加载主导访存):
$$
\text{AI} \approx \frac{2BDF}{2DF} = B
$$

**(1) Prefill 阶段**: 批量维度 $$B$$ 等于所有输入 tokens 的总数。对于一个有 512 个 prompt tokens 的请求, $$B = 512$$, AI ≈ 512 FLOPs/Byte——远超 H100 的 Ridge Point 295。因此, **Prefill 几乎总是计算受限**。

**(2) Decode 阶段**: 批量维度 $$B$$ 等于并发请求数(每个请求每次仅生成 1 个 token)。要进入计算受限区域, 需要 $$B > 295$$——即同时服务 295+ 个并发请求。但在实践中:
- **KV Cache 容量限制**: 每个请求的 KV Cache 持续增长。假设每个请求的序列长度为 4096, 使用 Llama-3-70B(80 层, 8 个 KV heads, head_dim=128): 单个请求的 KV Cache ≈ $$2 \times 80 \times 8 \times 4096 \times 128 \times 2 \text{ bytes} = 1.25 \text{ GB}$$。50 个请求就需要 62.5 GB——几乎占满 H100 的 80 GB 显存。
- **请求到达的统计特性**: 请求到达是随机的, 等待 295 个请求凑满一个批次会导致不可接受的延迟。

因此, **Decode 几乎总是访存受限**——GPU 的大量计算单元在等待数据从 HBM 加载, 这是推理优化的核心瓶颈。

**系统设计的推论**:

| 维度 | Prefill (Compute-Bound) | Decode (Memory-Bound) |
|:---|:---|:---|
| 瓶颈 | 计算吞吐量 | 内存带宽 |
| 优化目标 | 最大化 FLOPs 利用率 | 最小化内存访问 |
| 策略 | Tensor Core 利用, Kernel Fusion | 批量增大, 量化, KV Cache 压缩 |
| 并行化 | 张量并行(TP)分摊计算 | 增大批量受限于 KV Cache |

### 5.6.4 GPU 编程模型: CUDA 与 Triton

**CUDA**(Compute Unified Device Architecture, 2006 年发布)是 NVIDIA 的底层 GPU 编程模型。它扩展了 C++, 允许程序员:
- 定义**内核函数**(`__global__`): 在 GPU 上并行执行的函数
- 管理**内存**: `cudaMalloc`, `cudaMemcpy`(HBM 分配与 Host-Device 传输)
- 控制**线程层次**: `threadIdx`, `blockIdx`, `blockDim`, `gridDim`
- 使用**共享内存**: `__shared__` 关键字
- 同步线程: `__syncthreads()`

CUDA 提供了对 GPU 硬件的最细粒度控制——程序员需要显式管理数据在 HBM、L2、L1/Shared Memory 和寄存器之间的流动。编写高效的 CUDA 内核需要深入理解: Warp 调度、Bank Conflict、内存合并(Memory Coalescing)、占用率等底层细节。

**Triton**(OpenAI 发布, 2021 年)是一个更高层次的 GPU 编程语言, 用 Python 编写, 将内核编译为优化的机器码。Triton 的核心理念是:

- **块级编程**: 程序员在"块"(block)级别编写代码, 而不是在单个线程级别。Triton 编译器自动处理块内的并行化。
- **自动 Tiling**: Triton 编译器可以将大矩阵操作自动分块为适合共享内存的小块。
- **自动内存合并**: Triton 编译器自动优化内存访问模式以保证合并。
- **自动优化**: Triton 编译器可以自动选择最佳的 Tile 大小、展开因子等。

Triton 在灵活性和性能之间取得了很好的平衡——相比手写 CUDA(最灵活但最复杂), 相比 `torch.compile`(最自动但可能不如手写优化), Triton 允许程序员在"硬件的思维模式"下编程, 同时省去许多底层细节。

## 5.7 GPU 性能优化: 六大核心技术

### 5.7.1 控制流分歧 (Control Divergence)

GPU 以 Warp(32 线程)为单位执行。如果 Warp 中的线程因条件分支而走了不同的路径:

```cuda
if (threadIdx.x % 2 == 0) {
    // Path A: 偶数线程
} else {
    // Path B: 奇数线程
}
```

两条路径会被**串行化执行**: 偶数线程先执行 Path A(奇数线程空闲), 然后奇数线程执行 Path B(偶数线程空闲)。在最坏情况下(所有 32 个线程各自走不同路径), 执行时间变为串行的 32 倍。

**缓解策略**:
- 将数据重组为 warp-uniform 的访问模式(同一 Warp 内所有线程在相同分支)
- 避免基于 threadIdx 的条件分支
- 尽可能使用掩码操作(如 `__ballot_sync`, `__shfl_sync`)替代分支

### 5.7.2 低精度计算与量化

前面 5.4 节已经详细讨论了精度模式。这里从性能优化的角度做补充总结:

- **FP16/BF16 训练**: 前向和反向传播使用 FP16/BF16, 权重更新使用 FP32 主副本(Mixed Precision Training)。自动混合精度(AMP, Automatic Mixed Precision)在 PyTorch 中仅需一行 `torch.cuda.amp.autocast()`。
- **FP8 训练**: H100+ 原生支持。前向用 E4M3, 反向用 E5M2。需要重新设计训练的数值稳定性策略。
- **INT8/INT4 推理**: 权重量化为 INT8/INT4, 激活可以是 FP16。Tensor Core 的 INT8 吞吐量是 FP16 的 2 倍。

**量化实战提示**: 对于 LLM 推理, 量化收益远大于训练。这是因为 Decode 阶段是访存受限的——将权重从 FP16 量化到 INT4, 每次前向传播从 HBM 读取的数据量直接减少 4 倍, AI 等比例增加 4 倍, 性能提升立竿见影。

### 5.7.3 算子融合 (Operator Fusion)

将 GPU 想象为一个工厂: 输入来自仓库(显存), 在工厂中加工(计算)。现代 GPU 的计算能力飞速增长, 但"仓库到工厂"的运输带宽增长缓慢。算子融合的核心思想是: **尽可能在工厂内完成所有加工, 减少来回运输的次数**。

**朴素方式**(非融合): 每做一个操作就把数据送回显存, 再从显存读取给下一个操作。例如计算 $$\sin^2 x + \cos^2 x$$:
1. 加载 x → 计算 sin(x) → 写回 sin(x)
2. 加载 sin(x) → 计算 sin²(x) → 写回 sin²(x)
3. 加载 x → 计算 cos(x) → 写回 cos(x)
4. 加载 cos(x) → 计算 cos²(x) → 写回 cos²(x)
5. 加载 sin²(x) 和 cos²(x) → 计算加法 → 写回结果

总共 **5 次 CUDA 内核调用**, 每次都需要往返 HBM。

**融合方式**: 所有 5 个逐点操作融合为一个 CUDA 内核调用。数据只从 HBM 读取一次(x), 内部完成所有中间计算(中间结果保存在寄存器中), 最后只写回一次(最终结果)。内存访问从 $$5 \times (\text{读} + \text{写})$$ 减少为 $$1 \times \text{读} + 1 \times \text{写}$$——减少了 **80% 的带宽消耗**。

PyTorch 2.0 的 `torch.compile` 可以自动执行此类"简单"融合。但对于涉及不同维度变换的复杂融合(如 Attention 中的 softmax + 矩阵乘法), 需要手写融合内核。

### 5.7.4 重计算 (Recomputation / Activation Checkpointing)

反向传播需要前向传播中保存的激活值来计算梯度。但存储所有层的激活值会消耗大量显存——对于 Transformer 模型, 激活值可能比权重占用更多显存(特别是长序列时)。

**核心洞察**: 丢弃激活值并在反向传播时重新计算它们, 有时比存储并访问它们更高效。

考虑堆叠 3 个 Sigmoid 的简单例子: $$y = \sigma(\sigma(\sigma(x)))$$

**存储所有激活**: 8 次内存读/写(每层: 读输入 + 写输出, 外加反向传播时的额外读取)。极低的算术强度——几乎每字节数据上只有 1-2 FLOPs。

**丢弃并重计算**: 只存储第一层的输入 x(1 次写), 反向传播时从 x 重新计算中间激活值(每层仅需 1 次前向重计算)。总内存访问减少为原方案的 **5/8**。

**在 Transformer 中的实践**: PyTorch 的 `torch.utils.checkpoint`(activation checkpointing)会在前向传播中丢弃中间激活, 反向传播时重新执行被丢弃的部分。对于 175B GPT-3 的训练, 这是绝对必须的——否则激活值就会占满所有 GPU 显存。

**最优重计算策略**: 不是"全存"或"全丢"的二元选择, 而是一个图论中的**最小割问题**(Min-Cut Problem)。AOTAutograd(PyTorch)使用最小割算法来决定哪些激活值值得保存、哪些值得重计算——该算法权衡的是"保存的显存成本"vs"重计算的计算成本"。

### 5.7.5 内存合并 (Memory Coalescing)

DRAM(包括 HBM)以**突发模式**(Burst Mode)访问: 每次读取返回连续的一段字节(通常 32、64 或 128 字节), 而不是单个字节。这是由 DRAM 的物理结构决定的——从存储单元阵列中"感觉"(Sense)一行(数千字节), 然后通过列选通选择具体的数据。

**内存合并**指的是: 如果一个 Warp 中所有线程访问的地址都落在同一个突发(Burst)范围内, 这些访问可以合并为一次 DRAM 读取, 然后分发到各个线程。如果这些地址分散在多个 Burst 中, 则需要多次读取——性能严重下降。

**矩阵乘法的合并实例**:

对于行主序(Row-Major)矩阵 $$A[M, K]$$:
- 线程沿**列**方向访问(即 $$A[0, 0], A[1, 0], A[2, 0], \ldots$$): **不合并**——相邻线程的访问地址间隔 $$K$$ 个元素, 跨越多个 Burst
- 线程沿**行**方向访问(即 $$A[0, 0], A[0, 1], A[0, 2], \ldots$$): **合并**——相邻线程访问连续的内存地址, 落在同一 Burst 中

这就是为什么矩阵乘法中**内存布局**(行主序 vs 列主序)和**线程映射**需要配合设计——正确的组合可以最大化内存合并, 从而充分利用 HBM 带宽。

### 5.7.6 分块 (Tiling)

Tiling 是 GPU 编程中最重要的优化技术, 也是前面多项技术的集大成者。

**问题**: 在朴素矩阵乘法中, 每个输入元素被重复从全局内存读取 $$N$$ 次(其中 $$N$$ 是矩阵维度)。对于 $$4096 \times 4096$$ 的矩阵乘法, 每个元素被读取 4096 次——全局内存访问总量为 $$\sim 4096 \times 3 \times 4096^2 \times 2 \text{ bytes}$$, 远超 HBM 带宽能力。

**Tiling 的解决方案**: 将大矩阵切成小块(Tile), 每次将一个 Tile 加载到共享内存中:

1. 加载 $$M_{\text{tile}}$$ 和 $$N_{\text{tile}}$$ 到共享内存
2. 在共享内存中计算部分和(Partial Sums)
3. (完成一个 Tile)
4. 加载下一个 Tile 到共享内存
5. 继续计算部分和, 与之前的结果累加
6. ...

**收益分析**:
- 每个输入元素从全局内存读取 $$\frac{N}{T}$$ 次(而不是 $$N$$ 次), 其中 $$T$$ 是 Tile 的宽度——减少了 $$T$$ 倍的全局内存访问
- 在共享内存内部, 每个元素被读取 $$T$$ 次(每个 Tile 内的所有线程重复使用)——但共享内存的带宽是 HBM 的约 10 倍
- 同时可以实现内存合并: Tile 的加载可以将 Warp 的访问合并为少量的 Burst 读取

**Tiling 的复杂度**:

1. **Tile 大小选择**: 并非越大越好
   - 受限于共享内存大小(每 SM 仅约 164-228 KB)
   - 受限于占用率(更大的 Tile 使用更多共享内存, 限制了 SM 上可驻留的线程块数量)
   - 受限于矩阵维度(如果矩阵维度不能被 Tile 大小整除, 边缘 Tile 利用率低)

2. **内存对齐**: 如果 Tile 的起始地址不是 Burst 边界对齐的, 合并访问可能失效。特别是当矩阵维度是"非友好"数字时(如质数维度 1793), 内存对齐变得困难——有时需要使用 Padding 来强制对齐。

3. **Wave 量化(Wave Quantization)**: 由于 Tile 大小和 SM 数量的相互作用, 性能会出现周期性的波动。例如 A100 使用 256×128 的 Tile 做矩阵乘法。对于 1792×1792 的矩阵, 需要 $$\frac{1792}{256} \times \frac{1792}{128} = 7 \times 14 = 98$$ 个 Tile——恰好可以被 108 个 SM 一轮消化。但对于 1793×1793, 需要 $$8 \times 15 = 120$$ 个 Tile——超过了 108 个 SM, 需要两轮执行, 其中一轮大部分 SM 空闲。这就是"矩阵神秘学"的幕后真相: **为什么大矩阵有时比小矩阵快**——大矩阵的 Tile 数量恰好能被 SM 数量整除, 实现了完美的 Wave 利用率。

**综合**: Tiling 之所以是"大技术", 是因为它同时解决了多个层面: 全局内存访问减少(降低带宽压力)、共享内存重用(利用高速片上 SRAM)、内存合并(高效利用 HBM 的突发模式)——三者叠加, 可以实现理论带宽的 80-90% 利用率。

## 5.8 案例分析: FlashAttention 的硬件感知设计

### 5.8.1 背景: 注意力机制的计算与访存

Transformer 的标准缩放点积注意力(Scaled Dot-Product Attention)定义如下:

$$
\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V
$$

其中 $$Q, K, V \in \mathbb{R}^{N \times d}$$, $$N$$ 为序列长度, $$d$$ 为头维度。

**朴素实现的访存分析**:

1. 计算 $$S = QK^T$$: 从 HBM 读取 $$Q, K$$, 写回 $$S$$(矩阵乘法, $$N \times N$$ 输出)
2. 计算 $$P = \text{softmax}(S)$$: 读取 $$S$$, 写回 $$P$$(逐元素操作 + 归约)
3. 计算 $$O = PV$$: 读取 $$P, V$$, 写回 $$O$$(矩阵乘法)

**关键问题**: 中间矩阵 $$S$$ 和 $$P$$ 的大小是 $$O(N^2)$$——对于长序列, 这两个矩阵主导显存占用。以 $$N = 8192$$, $$d = 128$$, FP16 为例:
- $$Q, K, V$$ 各占 $$8192 \times 128 \times 2 = 2 \text{ MB}$$
- $$S$$ 和 $$P$$ 各占 $$8192 \times 8192 \times 2 = 128 \text{ MB}$$ ← 主导!

读写 $$S$$ 和 $$P$$ 消耗的 HBM 带宽远超读写 $$Q, K, V$$。而 $$S$$ 和 $$P$$ 是**中间结果**——它们在计算后立刻被下一个操作使用, 然后被丢弃。将这两个大矩阵写入 HBM 再读回, 纯粹是一种浪费——HBM 的有限带宽被大量不必要的读写占满, 计算单元只能等待。

这就是 FlashAttention 要解决的核心问题: **避免将 $$O(N^2)$$ 的中间矩阵写入 HBM**。

### 5.8.2 三项核心技术的融合

FlashAttention 的成功来自三项技术的巧妙组合:

**技术一: Tiling — 分块计算 KQV 矩阵乘法**

将 $$Q, K, V$$ 沿序列维度切分为 Tile, 每次只计算一个 Tile 的输出。这本质上是标准的矩阵乘法 Tiling——但问题来了: softmax 需要跨整个 $$N$$ 维度归一化, 而我们现在只有一个 Tile 的信息。

**技术二: 在线 Softmax (Online Softmax)**

标准的 softmax 定义:
$$
\text{softmax}(x_i) = \frac{e^{x_i}}{\sum_{j} e^{x_j}}
$$

需要先扫描全部元素计算最大值和归一化因子, 再做逐元素计算——这就是通常需要将整个矩阵写入 HBM 的原因。

在线 Softmax(来自 Milakov & Gimelshein, 2018)提供了一个增量计算的方案:

1. 维护当前见到的**最大值** $$m$$ 和**归一化因子** $$\ell$$
2. 当新的一批数据到达时:
   - 更新最大值: $$m_{\text{new}} = \max(m, \max(\text{new\_data}))$$
   - 重新缩放旧的归一化因子: $$\ell \leftarrow \ell \times e^{m - m_{\text{new}}}$$
   - 累加新的指数和: $$\ell \leftarrow \ell + \sum e^{\text{new\_data} - m_{\text{new}}}$$

这种"伸缩和"(Telescoping Sum)的更新方式允许我们逐 Tile 地计算 softmax, 不需要一次性看到全部数据。

**技术三: 内核融合 — 将所有操作合并为一个 CUDA 内核**

将上述所有步骤——Tile 加载、内积计算、指数运算、在线 softmax 更新、Tile 输出——全部融合为一个 CUDA 内核调用。数据从 HBM 读取一次, 中间结果全部保存在寄存器和共享内存中, 仅最终结果写回 HBM。

### 5.8.3 前向传播的执行流程

FlashAttention 的前向传播可以概括为以下步骤(来自 Dao et al., 2023):

1. **外循环(沿 $$K, V$$ 的 Tile)**: 遍历 $$K$$ 和 $$V$$ 的 Tile
   - 从 HBM 加载一个 $$K, V$$ 的 Tile 到共享内存

2. **内循环(沿 $$Q$$ 的 Tile)**: 对每个 $$Q$$ 的 Tile
   - 从 HBM 加载一个 $$Q$$ Tile 到共享内存
   - 在共享内存中计算 $$S_{\text{tile}} = Q_{\text{tile}} \times K_{\text{tile}}^T$$(部分内积)
   - 在共享内存中计算 $$\exp(S_{\text{tile}})$$(融合指数运算)
   - 使用在线 softmax 更新保持的最大值和归一化因子
   - 重新缩放之前累加的输出(基于最大值的变化)
   - 累加当前 Tile 对输出 $$O_{\text{tile}}$$ 的贡献: $$O_{\text{tile}} \leftarrow O_{\text{tile}} + P_{\text{tile}} \times V_{\text{tile}}$$
   - 最终 $$O_{\text{tile}}$$ 写回 HBM

**关键**: 整个过程中, $$N \times N$$ 大小的中间矩阵 $$S$$ 和 $$P$$ **从未完整存在于 HBM 中**——它们以 Tile 为单位短暂存在于共享内存, 计算完立刻被下一个 Tile 覆盖。

**关于反向传播**: FlashAttention 的反向传播采用了额外的设计——在反向传播中重计算 Tile(即 Activation Checkpointing), 而不是存储前向传播中的所有中间结果。这进一步减少了显存占用。

### 5.8.4 FlashAttention 的性能分析

FlashAttention 为什么快? 从 Roofline 模型的角度分析:

**朴素注意力**:
- HBM 读写: $$O(N^2)$$ 字节(中间矩阵 $$S$$ 和 $$P$$ 是瓶颈)
- 计算: $$O(N^2 \cdot d)$$ FLOPs
- 算术强度: $$\frac{O(N^2 d)}{O(N^2)} = O(d) \ll AI_{\text{ridge}}$$ 对于典型 $$d = 128$$
- **访存受限** — 大量时间花在等待 HBM 传输

**FlashAttention**:
- HBM 读写: $$O(N^2 d^2 / M)$$ 字节(其中 $$M$$ 是共享内存大小, Tile 越大访存越少)
- 计算: $$O(N^2 d)$$ FLOPs(相同)
- 算术强度: 提升了 10-20 倍(取决于 Tile 大小和序列长度)
- **从访存受限转变为计算受限**(或接近 Ridge Point)

**数值效果**(引自 Dao et al. 2023, A100 GPU):
- 训练速度: 2-4 倍加速(长序列时效果更显著)
- 显存使用: 节省 10-20 倍(中间矩阵不再占满 HBM)
- 允许训练更长的序列长度: 从 1K → 8K → 64K tokens

**FlashAttention 的局限性**:
- 要求序列长度 $$N$$ 能被 Tile 大小整除(或损失部分利用率)
- 对 attention mask 的支持依赖于具体的 kernel 实现
- 反向传播的重计算增加了少量计算开销(但远小于节省的带宽)

### 5.8.5 从 FlashAttention 学习的系统设计方法论

FlashAttention 不仅是注意力机制的一个优秀实现, 更代表了一种**硬件感知算法设计**(Hardware-Aware Algorithm Design)的方法论:

1. **识别真正的瓶颈**: $$O(N^2)$$ 的中间矩阵不是"需要的", 而是算法实现方式的产物。从数学上看, softmax 的分母可以用在线方式计算——改变算法可以消除"看似不可避免"的中间结果。

2. **匹配算法到硬件层次**: 将计算结构(Tiling)与硬件结构(SM 的共享内存)对齐。Tile 大小应根据共享内存容量和 Warp 大小选择——这是"硬件倒推算法"的典型体现。

3. **融合以减少数据移动**: 多个数学运算可以在同一个内核中完成, 中间结果留在寄存器和共享内存中。数据移动(而非计算)是 GPU 优化的首要关注点。

4. **精度与性能的协同**: FlashAttention 在使用 FP16/BF16 进行计算时, 通过在线 softmax 维护了数值稳定性——证明了"低精度 + 稳定算法"的组合可以同时获得高性能和高稳定性。

**推广**: 这些方法论的适用范围远不止于注意力机制。任何存在"临时大张量"的计算(如 MoE 的 gating 计算、KV Cache 的管理、序列并行中的通信)都可以从类似的角度重新审视——是否有比"写出再读回"更好的计算组织方式?

## 5.9 本章总结

**核心要点回顾**:

1. **GPU 是大规模并行的吞吐量引擎**: 数千个 ALU 通过 SIMT 模型执行相同的指令, 零开销 Warp 切换隐藏内存延迟, SM 中的共享内存在线程块内提供高速数据共享。

2. **Tensor Core 通过专用硬件将矩阵乘法加速 10 倍以上**: 从 FP16 到 MXFP4, 精度模式的选择直接影响内存占用和算术强度, 低精度计算是 LLM 从 H100/H200 获得性能的核心手段。

3. **TPU 的脉动阵列与 GPU 的 SIMT 是两种互补的设计哲学**: GPU 更灵活(非 MatMul 操作也可高效执行), TPU 的 MatMul 密度更高(但不够灵活)。两者的系统级差异(网络互联、编程模型)在实践中同样重要。

4. **Roofline 模型提供了评判"快慢"的严格框架**: 算术强度(AI)决定了一个算子在给定硬件上是计算受限还是访存受限。对于 H100, Ridge Point = 295 FLOPs/Byte——低于此值的操作本质上是"等待数据", 不是"等待计算"。

5. **GPU 优化的六大技术是分层的**: 低精度减少数据量、合并提高带宽利用率、融合减少 HBM 往返、Tiling 将数据移到共享内存、重计算用计算换内存、控制流分歧优化减少 Warp 序列化——它们可以在同一天的内核中叠加使用。

6. **FlashAttention 是硬件感知算法设计的典范**: 通过 Tiling + 在线 Softmax + 内核融合三项技术的组合, 避免了将 $$O(N^2)$$ 的中间矩阵写入 HBM, 将注意力计算从访存受限转变为接近计算受限——实现了 2-4 倍的端到端加速。

**反思**: 回到本章开篇的命题——"不理解硬件就无法写出高效的 ML 代码"。CUDA/Triton 内核编程曾经是系统工程师的专属领域, 但随着 LLM 的规模增长, 硬件效率的每个百分点都在转化为数百万美元的成本差异。FlashAttention、vLLM 的 PagedAttention、DeepSeek 的 MLA(Multi-head Latent Attention)都是"算法-硬件协同设计"的产物。理解 GPU 和 TPU 的工作原理, 不是为了成为内核工程师, 而是为了在更高的抽象层次上做出正确的系统设计决策——这是每一位 ML 系统研究者和工程师的必修课。
