# 第11章 数据处理 (Data for Language Models)

> 译自 CS 336 Language Modeling from Scratch, Lectures 13-14: Data for Language Models (Tatsu Hashimoto)
> 数据是大语言模型的"燃料"。模型架构决定了能力的上限，而数据的质量、多样性和规模决定了模型能够多接近这个上限。本章从数据来源、处理流水线、去重、混合策略、合成数据到数据污染与法律伦理，系统梳理 LLM 数据工程的全链路。


---

## 11.1 引言：数据——大语言模型的"燃料"

### 11.1.1 本章目标

在 Transformer 架构和训练技术日趋成熟的今天，**数据已成为 LLM 竞争力的核心差异化因素**。2024 年的一个共识是：不同实验室之间的模型架构（Pre-Norm + RMSNorm + SwiGLU + RoPE + GQA）已高度趋同，但数据配方的差异导致了模型能力的显著分化。Meta 的 LLaMA 系列、TII 的 Falcon、以及 DeepSeek 的出色表现，很大程度上归功于它们在数据工程上的创新投入。

本章的核心问题是：**构建一个高质量 LLM 预训练语料库需要做什么？每个步骤为什么重要？数据质量到底有多关键？**

### 11.1.2 数据工程的四层视角

我们可以将 LLM 数据工程分为四个层次：

```
第4层  数据混合与课程        ← 不同来源的配比、训练阶段的数据调度
第3层  数据增强与合成        ← Self-Instruct、蒸馏、课程学习
第2层  数据清洗与去重        ← 质量过滤、精确/模糊去重、语言检测
第1层  原始数据获取          ← Common Crawl、Wikipedia、Books、Code
```

本章将沿着从底层到顶层的顺序展开，最后讨论数据污染与法律伦理两个横切关注点。

### 11.1.3 一个核心张力：质量 vs 数量

在讨论任何一个具体技术之前，有必要先阐明贯穿全章的核心张力：我们既需要海量数据来满足模型容量的需求（Chinchilla 定律），又需要数据在内容和格式上足够"干净"以保证模型学习的有效性。这两者是冲突的——最大的开源语料库（Common Crawl）恰好也是"最脏"的。如何处理这个张力，是数据工程的根本命题。

## 11.2 数据来源全景：从互联网到代码仓库

### 11.2.1 主要数据来源概览

现代 LLM 预训练数据通常来自以下六大类别：

| 来源 | 典型规模 | 主要价值 | 关键挑战 |
|------|---------|---------|---------|
| **Common Crawl** | ~200B+ pages, PB 级 | 规模最大，覆盖广泛 | 噪声极高，需大量清洗 |
| **Wikipedia** | ~6M 英文文章 | 高质量、结构化、多语言 | 规模有限，单次抓取即足够 |
| **Books** | ~100K-200K books (Books3, PG-19) | 高质量长文本、叙事结构 | 版权灰色地带，覆盖有限 |
| **Web News / Curated Web** | C4, OpenWebText2, FineWeb | 中等质量，广泛覆盖 | 过滤偏差（新闻偏向正式语体） |
| **Code** (GitHub, StackOverflow) | The Stack (~6TB), StarCoder data | 推理能力的核心来源 | 许可证、低质量代码、安全漏洞 |
| **Academic Papers** (ArXiv, PubMed) | S2ORC, Pile-CC | 高质量、结构化推理 | 领域偏见，数学公式难以 tokenize |
| **Conversational** (Reddit, forum) | OpenWebText, Pile subsets | 对话自然度 | 毒性内容、偏见 |

### 11.2.2 Common Crawl：数据的主粮

Common Crawl 是一个非营利组织，每月抓取整个互联网（约 30 亿页面），并以 WARC 格式公开发布。它几乎是所有现代 LLM 的最大单一数据来源——从 GPT-3、T5、Gopher 到 LLaMA、Falcon、Gemma，无一例外。

**为什么 Common Crawl 如此重要？**

1. **规模无与伦比**：仅一次月度抓取就包含约 200-300TB 的原始 HTML。经过过滤后仍可产生数万亿 token 的训练数据——远超任何其他开源来源。
2. **内容多样性**：覆盖新闻、博客、论坛、商品页面、学术主页、政府网站、个人主页……这种多样性对于模型的泛化能力至关重要。
3. **可自由获取**：作为公共资源，Common Crawl 推动了学术界和开源社区的数据创新（如 C4、RefinedWeb、DCLM、FineWeb 等数据集均基于 Common Crawl 构建）。

**但 Common Crawl 的质量挑战同样巨大**：
- 大量页面是 SEO 垃圾、广告、导航页面、自动生成内容。
- HTML 标签、JavaScript 代码、CSS 嵌入在文本中。
- 内容重复率极高（不同 URL 指向相同或高度相似的内容）。
- 多语言混合（一个页面可能包含多种语言）。

正因为如此，Common Crawl 的使用高度依赖下游的处理流水线——这也是 11.3 节要讨论的核心内容。

### 11.2.3 Wikipedia：高质量的基石

维基百科虽然规模有限（英文版约 60 亿词），但几乎出现在所有 LLM 的训练数据中。原因有三：

- **事实准确性**：集体编辑和引用机制使 Wikipedia 成为事实核查的良好基准。
- **格式规范**：Markdown/Wikitext 格式相对干净，段落结构清晰。
- **多语言覆盖**：300 多种语言的 Wikipedia 为多语言模型提供天然种子。

许多工作流会给 Wikipedia 更高的重要性权重（upsampling），例如 2-10 倍。甚至有些模型在训练后期增加 Wikipedia 的比例，作为一种"课程学习"策略——先用大量网络文本建立语感，再用高质百科巩固事实准确性。

### 11.2.4 Books：长文本与深层叙事

书籍数据（如 Books3、Project Gutenberg、PG-19）提供了网络文本和百科无法替代的价值：

- **长程依赖**：一本书中的情节和角色跨越数万甚至数十万 token，对模型的长上下文建模能力是极好的训练信号。
- **复杂语法和叙事结构**：书籍的句法复杂度远高于网络文本。研究表明，书籍数据的加入显著提升了模型在长文本生成和推理任务上的表现。
- **文化知识的载体**：经典文学、科学著作、历史文献包含大量隐含的文化知识。

然而，**版权问题是书籍数据的巨大阴影**。Books3 数据集（源自 ThePirateBay 的 Bibliotik 私种）已被许多商业实验室从训练数据中移除。PG-19（Project Gutenberg 的公版书）是完全合法的替代方案，但局限在 1919 年之前出版的书籍。

### 11.2.5 Code：推理能力的秘密武器

代码数据的作用远超"教会模型写代码"本身。这是 LLM 研究中的一个重要发现——代码训练可以**跨任务提升模型的推理能力**。

**为什么代码数据如此特别？**

1. **形式语言的逻辑结构**：与自然语言不同，代码具有精确的语法和确定的执行语义。训练模型预测下一个代码 token 类似于"软编译器"的学习过程，强化了模型的逻辑推理能力。
2. **长距离依赖**：一个函数的调用可能距离其定义数千行。这种长程因果链的训练信号远强于自然语言文本。
3. **结构化思考的迁移**：研究发现，经过代码训练的模型在数学推理、多步规划、逻辑推理等任务上的表现显著优于仅训练自然语言的模型——即使这些任务本身不涉及代码。

**主要代码来源**：

- **GitHub**: The Stack v1/v2（由 BigCode 项目整理）是最大的开源代码数据集，包含约 6TB 的去重代码和宽松许可证过滤。
- **StackOverflow**: 问答形式的代码数据，提供了代码与其自然语言解释的配对——这对指令遵循能力特别有价值。
- **Jupyter Notebooks**: 代码、文本、输出的交织，是训练代码理解和交互能力的理想格式。

**代码数据的特殊挑战**：
- **许可证合规**：并非所有 GitHub 仓库的许可证都允许用于 AI 训练。The Stack 项目通过 opt-out 机制允许开发者移除自己的代码。
- **安全漏洞**：训练数据中的代码可能包含安全漏洞（SQL 注入、缓冲区溢出等），模型可能学会并复现这些漏洞。
- **质量参差不齐**：GitHub 上存在大量教学示例、废弃项目和不完整的代码片段。

### 11.2.6 学术论文：结构化知识的富矿

ArXiv、PubMed Central 等学术平台提供了高度结构化、引用密集的文本。S2ORC（Semantic Scholar Open Research Corpus）整理了超过 1.36 亿篇学术论文。

**主要价值**：
- 科学推理的严密结构：摘要 - 引言 - 方法 - 实验 - 结论。
- LaTeX 格式的数学公式为训练数学推理提供了监督信号。
- 引用网络可用于构建知识图谱增强的训练目标。

**局限性**：
- 领域偏见（物理、CS、数学过度代表）。
- $$L^AT_EX$$ 语法与自然文本的混合使得分词变得困难。
- 学术写作风格与实际应用场景（对话、客服、创意写作）差距很大。

### 11.2.7 对话与论坛数据

Reddit、Stack Exchange、Quora 等平台的对话和问答数据提供了自然的对话结构。OpenWebText2 是此类数据的代表——虽然名为 "Web Text"，但它主要通过 Reddit 的 outbound link 构建，实际上是对高赞 Reddit 帖子链接页面的收集。

Pushshift Reddit 数据集（已被移除公共访问）曾广泛用于 Pile 等数据集。替代方案包括 OpenWebText2 和 Common Crawl 的子集（筛选 Reddit 和社交媒体链接）。

**数据来源小结**：不同来源的数据在质量、规模、风格和可用性上形成互补。核心挑战不在于"获取"，而在于如何将这些异构的原始数据转化为干净、去重、均衡的训练语料——这正是接下来三节的主题。

## 11.3 Web 数据处理流水线

### 11.3.1 流水线全景

从原始 HTML 到可用于训练的 token 序列，需要经过一条多阶段的处理流水线。以下是一个典型的 Web 数据处理流水线（基于 C4、RefinedWeb、FineWeb、DCLM 等主流方案）：

```
原始 WARC/WET
    │
    ▼
[1] HTML 提取（文本抽取）
    │
    ▼
[2] 样板内容移除（Boilerplate Removal）
    │
    ▼
[3] 语言检测（Language Detection）
    │
    ▼
[4] 质量过滤（Quality Filtering）
    │
    ▼
[5] 去重（Deduplication）──→ 见 11.4 节
    │
    ▼
[6] PII 脱敏 / 内容安全过滤
    │
    ▼
干净文本 → Tokenization → 训练
```

每个阶段都可以选择不同的工具和策略，它们的组合决定了最终语料的质量和产出率（yield）——即从原始数据中保留了多大比例的文本。

### 11.3.2 HTML 提取（HTML Extraction）

Common Crawl 以 WARC（Web ARChive）格式存储数据，每条记录包含完整的 HTTP 响应——HTML 标签、CSS、JavaScript、以及文本内容。第一步是从中提取纯文本。

**常用工具**：

- **trafilatura**: 一个专为学术文本分析设计的 Python 库，在提取正文内容方面表现优异。适合新闻、博客、文章类页面。
- **jusText**: 基于启发式算法（链接密度、文本密度）移除样板内容，同时提取正文。是 Common Crawl 文本提取的常见选择。
- **html-text**: 一个简洁的工具，通过解析 HTML 标签结构提取可见文本。
- **BeautifulSoup + 自定义规则**: 许多团队编写自己的提取规则，针对特定类型的页面（如维基、论坛）进行优化。

**核心挑战**：
- 页面结构多样性巨大——同一提取器在一个新闻网站上效果完美，在论坛页面上可能完全失败。
- 提取后文本中的换行符、空白字符、特殊字符（如 `&amp;`、`&#160;`）需要进一步规范化。
- 对于 JavaScript 渲染的页面（SPA），静态 HTML 提取可能只能拿到空壳——这也是 Common Crawl 的一个根本局限。

**文本产出率**：一个典型的 Common Crawl dump 在 HTML 提取后，约 30-50% 的页面能产生有意义的文本（长度 > 100 字符）。其余页面主要是导航页、404 错误页、JavaScript 占位页面和纯图片/视频页面。

### 11.3.3 样板内容移除（Boilerplate Removal）

HTML 提取后，文本中仍可能残留大量"样板内容"——即与页面主题内容无关的重复性文本。

**常见的样板内容包括**：
- 导航菜单文字
- 页脚信息（"All rights reserved"、"Cookie Policy"）
- 侧边栏内容
- 广告文案
- 评论区模板
- JavaScript 残留和 CSS 残留
- 自动生成的分页导航文字（"Page 1 of 200"）

**常用方法**：

1. **启发式规则**：基于文本密度（正文地区的文本占比高）、链接密度（导航区链接多）、重复模式（同一域名下多页面的共同子串）进行分类。jusText 和 trafilatura 内置了此类规则。
2. **DOM 子树分析**：利用 HTML DOM 树结构，识别出包含连续长文本块的最小 DOM 节点。Dragnet（2014）和 boilerpy3 是此类方法的代表。
3. **基于分类器的方法**：训练一个 BERT 级别的分类器，识别每个文本段是"正文"还是"样板"。这种方法泛化能力更强，但计算开销也更大。

**为什么这个步骤重要？**

样板内容在整个 Common Crawl 中高度重复——每个使用相同 CMS（如 WordPress）的网站都有相似的页脚和导航文字。如果不移除，这些重复文本会：
- 占据宝贵的 token 预算（在一个数万亿 token 的语料中，样板内容可能占 5-15%）。
- 让模型学会生成导航菜单和页脚文字——实用价值为零。
- 干扰去重算法的效果（内容不同但样板相同的页面被误判为相似）。

### 11.3.4 语言检测（Language Detection）

对于单语（通常是英语）模型，语言检测是过滤的一个关键步骤。Even a small percentage of non-target language data can significantly degrade performance on the target language。

**常用工具**：

- **fastText LangDetect** (Joulin et al., 2017): Meta 发布的快速线性分类器，支持 176 种语言。基于字符 n-gram 特征，每秒可处理数千条文本。
- **langdetect**: Google 的语言检测库移植版，基于朴素贝叶斯分类器，支持 55 种语言。
- **CLD2/CLD3 (Compact Language Detector)**: Google Chrome 中使用的语言检测器，速度极快，适合流式处理。
- **GLOTLID**: 更新的开源替代方案，支持更多小众语言。

**语言检测的常见陷阱**：

1. **混淆语言**：短文本（如标题）的语言特征不足，容易被误判。一般而言，文本长度至少需要 50-100 字符才能获得可靠的检测结果。
2. **代码与自然语言的区分**：代码文件可能被误判为"英语"（因为关键词和函数名多为英文）。需要使用专门的编程语言检测器进行区分。
3. **多语言页面**：一篇文章可能用英语撰写但包含法语引文。简单的"过滤非英语"策略会丢弃这些合法内容。常见的做法是：主语言概率 > 0.65-0.8 时判定为通过。
4. **语言边界**：挪威语 (no) 与丹麦语 (da)、捷克语 (cs) 与斯洛伐克语 (sk) 等相似语言对容易混淆。

**最佳实践**：在大型流水线中，语言检测通常在早期阶段进行（在昂贵的质量过滤之前），以尽早丢弃不相关语言的内容，节省后续步骤的计算资源。

### 11.3.5 质量过滤（Quality Filtering）

质量过滤是 Web 数据处理流水线中最关键也最复杂的环节。目标是从已提取的文本中，筛选出对语言模型训练有价值的"高质量"文本，丢弃低质量、无意义或有害的内容。

这是一个**多维度决策问题**：没有单一的"质量"定义，不同研究者和实验室对"高质量文本"的理解也可能不同。以下是从实际操作角度对质量过滤方法的分类。

#### 11.3.5.1 启发式过滤（Heuristic Filtering）

启发式过滤使用人工定义的规则来判断文本质量。这是最早、最简单、也是仍在广泛使用的方法。

**C4 数据集的启发式规则（Raffel et al., 2020）**：
1. 只保留以句号、感叹号、问号或引号结尾的行（去除列表和表格中的行碎片）。
2. 丢弃包含少于 5 个句子的页面。
3. 丢弃任何包含单词 "lorem ipsum" 的页面（排版占位文本）。
4. 丢弃任何包含花括号 `{}` 的页面（JavaScript/CSS 残留）。
5. 丢弃任何包含 "List of Dirty, Naughty, Obscene, or Otherwise Bad Words" 中词汇的页面。
6. 丢弃文档中超过 90% 的行长度介于 3-13 个 token 的页面（暗示是列表或诗歌）。

**其他常见启发式规则**：
- **重复 n-gram 比率**：如果文档中某个 5-gram/6-gram 重复出现超过一定比例（如 30%），判定为低质量（可能是机器生成或模板化的垃圾文本）。
- **词级困惑度（Word-level Perplexity）**：使用一个在干净文本上训练的小语言模型（如 KenLM 5-gram 模型），丢弃困惑度异常高的文档（暗示内容混乱或不连贯）。
- **特殊字符比率**：包含过高比例的特殊字符（`#`, `@`, `|`, `\`）的文本通常是格式化垃圾。
- **大写字母比率**：全部大写或超过 30% 大写字母的文本通常是无意义的。
- **停用词比率**：停用词占比过高（缺少实义词）或过低（实义词堆积但语法不通）都是低质量信号。
- **平均词长**：过长或过短的平均词长都暗示异常。

**启发式过滤的优缺点**：
- 优点：计算开销低、可解释性强、不需要标注数据。
- 缺点：规则是静态的，可能漏掉新类型的垃圾（如 LLM 生成的文本）；可能误杀合法的创意文本（如诗歌、代码文档）；规则之间可能相互冲突。

#### 11.3.5.2 基于困惑度的过滤（Perplexity-based Filtering）

**核心思想**：使用一个在"干净"文本上训练的语言模型，计算待评估文档的困惑度。困惑度过高的文档被认为"不正常"而被丢弃。

**具体实现**（以 CCNet 和 Wenzek et al., 2020 为例）：
1. 在 Wikipedia 或类似的高质量文本上训练一个轻量级语言模型（如 5-gram KenLM 模型）。
2. 对待评估文档计算逐句（或逐段）的困惑度分数。
3. 设置阈值（如困惑度中位数 + 1-2 个标准差），丢弃困惑度过高的文档。

**变体——"干净度"分数（DCLM, 2024）**：
DCLM 提出了一个更精细的困惑度过滤方法：同时训练两个 KenLM 模型——一个在高质量语料上、一个在 Common Crawl 的随机样本上。然后用这两个模型的困惑度差值作为"干净度"分数：
- 如果文档在"好模型"下的困惑度远低于"泛模型"下的困惑度，说明它更像高质量文本。
- 文档的"干净度" = PPL(common) - PPL(clean)，分数越高越好。

**困惑度过滤的优缺点**：
- 优点：比简单启发式规则更细致，能捕捉文本的"人类写作自然度"。
- 缺点：训练语言模型本身需要高质量语料（鸡生蛋问题）；语言模型可能对特定领域（如医学、法律）的文本给出高困惑度，即使这些文本质量很高；计算开销显著高于启发式规则。

#### 11.3.5.3 基于分类器的过滤（Classifier-based Filtering）

**核心思想**：训练一个文本分类器，直接预测"这个文档是否值得用于训练"。这是近年来的主流趋势。

**代表性工作**：

- **FastText 质量分类器（Brown et al., 2020 / GPT-3 的过滤方法）**：使用在高质量语料（如 Wikipedia、Books、WebText）和 Common Crawl 随机样本上训练的 FastText 线性分类器，将 Common Crawl 中的每个文档分类为"高质量"或"低质量"。

- **FineWeb-Edu (2024)**：HuggingFace 的 FineWeb 项目训练了一个基于 Snowflake Arctic 嵌入的分类器，专门识别具有教育价值（edu value）的网页。它是 2024 年最成功的质量过滤工作之一，产出的 FineWeb-Edu 数据集经消融实验证明在相同数据量下性能显著优于其他 Common Crawl 派生数据集。

- **DCLM 的模型驱动过滤（2024）**：DCLM 使用 fastText 分类器 + 困惑度模型 + 启发式规则的组合，并通过消融实验（训练小型模型在所有过滤配置上评估）来选择最优过滤策略。

**基于分类器的过滤的关键实践**：

1. **标注数据**：需要高质量的"正样本"和"负样本"。正样本通常来自 Wikipedia、Books、Curated Web (如 Reddit 高赞链接)；负样本来自 Common Crawl 的随机（低质）样本。
2. **领域漂移**：分类器可能学会特定的表面特征（如"wiki"、"article"等词的存在），而非真正的文本质量。需要在训练数据和特征工程上进行防御。
3. **迭代过滤**：先粗过滤（启发式规则），再用分类器精细过滤——这样可以减少分类器需要处理的文本量，同时避免分类器在明显的垃圾文本上浪费容量。

### 11.3.6 内容安全过滤（Safety Filtering）

除了"质量"，还需要考虑"安全"。内容安全过滤移除的是：
- 仇恨言论、极端暴力内容
- 儿童安全相关违规内容（CSAM）
- 露骨色情内容
- 个人可识别信息（PII）——详见 11.8 节

**常用方法**：
- 基于关键字的列表匹配（毒性词表、CSAM 哈希值）。
- 基于分类器的毒性检测（如 Perspective API——尽管其主要用于评估，而非大规模过滤）。
- NSFW 图像/文本检测器（用于多模态数据的文本部分）。

安全过滤面临的核心矛盾：过度激进的过滤可能意外移除合法但敏感的文本（如新闻报道、历史文献、医学文本），导致模型产生"过滤偏差"（filtering bias）——对特定话题的知识缺失或偏见。

### 11.3.7 产出率与取舍

一个关键的实际问题是：**从 Common Crawl 开始，经历所有这些步骤后，还剩多少数据？**

| 数据集 | 原始来源 | 产出率 | 关键过滤策略 |
|--------|---------|-------|-------------|
| C4 | Common Crawl | ~3-5% | 启发式 + 脏词列表 |
| RefinedWeb (Falcon) | Common Crawl | ~5-7% | 启发式 + URL去重 + 文档级去重 |
| FineWeb | Common Crawl | ~8-10% | 启发式 + 教育分类器 |
| DCLM | Common Crawl | ~5-8% | 困惑度 + FastText 分类器 + 启发式 |
| Pile | 多源混合 | ~100% (精选) | 按来源单独处理 |

产出率越低，过滤越严格，训练数据的"平均质量"越高——但这不一定意味着最终模型更好。过度过滤会：
1. 减少数据多样性（幸存的数据偏向于某些领域的写作风格）。
2. 减少数据总量，可能低于模型容量的 Chinchilla 最优数据量。
3. 引入过滤器的偏见（例如，过度依赖一个特定分类器的判断标准）。

**CS 336 的洞察**：最优的过滤策略是在给定计算预算和数据量约束下的经验选择——不存在"完美的通用过滤器"。DCLM 的消融实验方法（为每种过滤配置训练一个小模型并评估下游性能）是当前最严谨的实践范式。

## 11.4 数据去重：精确、近似与 URL 级别

### 11.4.1 为什么去重如此重要？

数据去重是大语言模型数据工程中**性价比最高的单一操作**。以下是为什么去重能显著提升模型性能的深层原因：

**原因 1：消除记忆化，促进泛化**

当训练数据中包含大量重复（同一段维基百科文字在网络不同角落出现了 100 次），模型不必学会理解这段文字——它只需要"背下来"。这导致：
- 模型在训练 loss 上表现良好，但在未见过的数据上泛化能力下降。
- 模型倾向于逐字复述训练数据（memorization），而非提取可迁移的语义知识。
- 在生成任务中，过度记忆可能导致隐私泄露（复述训练数据中的个人信息）。

Lee et al. (2022) 的经典实验表明：在 C4 数据集上去重后，用相同数量的训练步骤，模型在下游任务上的性能**反而提升**——尽管实际可用的训练 token 数减少了。

**原因 2：防止数据泄漏到基准测试**

许多 NLP 基准数据集（如 LAMBADA、WinoGrande、HellaSwag）的文本源自网络。如果不去重，这些基准中的文本可能以几乎完全相同的形式出现在训练数据中。这不是"模型学会了"，而是"模型背过了"——导致基准评估无效（详见 11.7 节数据污染）。

**原因 3：改善训练效率**

重复数据意味着在给定的训练计算量下，模型实际遇到的**独特信息**更少。从信息论角度看，每个 epoch 中重复文本带来的新信息量接近于零。去重提高了每个训练 token 的"信息密度"。

**原因 4：缓解"过度训练"中的记忆化**

即使是单 epoch 训练（standard practice for LLMs），重复数据也可能导致某些段落被多次"看到"。随着模型容量增大和训练计算量增加，这种记忆化风险随之上升。去重是减少这种风险的直接手段。

### 11.4.2 URL 级去重

URL 去重是最简单、成本最低的去重层次。其基本操作是：对于指向相同 URL 的多个记录，只保留一个版本。

在 Common Crawl 中，同一个 URL 可能在不同月份的抓取中出现（同一篇博客文章被反复抓取）。URL 去重通常作为去重流水线的第一步，可以快速消除最明显的重复。

**局限性**：
- 同义词 URL（`example.com/p=1` vs `example.com/page?id=1`）指向同一内容，但 URL 不同。
- 通过负载均衡或 CDN 的镜像内容，URL 完全不同。
- 同一内容在不同平台的转载（同一篇文章发布在多个新闻网站上）。

URL 去重是必要但不充分的——它只能捕获最表面的重复。

### 11.4.3 精确去重（Exact Deduplication）

精确去重检查两个文档是否**完全相同**（或包含相同的大段内容）。

**标准方法**：
1. 对每个文档计算哈希值（如 SHA-256 或 MD5）。
2. 将哈希值排序，找出完全相同的文档对。
3. 在重复文档对中，保留一个，丢弃其余。

**细粒度的精确去重**：除了文档级去重，更细致的做法是在**子文档级别**（如段落、句子、或固定大小的 n-gram 窗口）进行精确去重。Min et al. (2024) 的 SubDup 工作证明了子文档级去重的有效性——许多"文档级"不同的页面，在段落级别是高度重复的（镜像、引用、转载）。

**精确去重的优缺点**：
- 优点：零信息损失（不会误除独一无二的内容），实现简单，成本可控。
- 缺点：只能捕获完全相同的文本。一个字符的差异（如不同网站的页眉/页脚）也能让精确去重失败。

### 11.4.4 近似（模糊）去重 — MinHash

近似去重（Near-Deduplication 或 Fuzzy Deduplication）旨在捕获"几乎相同"的文档——如同一篇文章的不同转载版本、同一模板生成的不同页面、或仅因广告插入而略有不同的相同内容。

**MinHash 算法**：

MinHash 是近似去重的标准工具（最早由 Broder, 1997 提出用于搜索引擎去重，后被广泛应用于 LLM 数据处理）。

核心原理：
1. **Shingling**：将文档转化为 n-gram 的集合（例如，一个文档表示为所有 5-gram 的集合）。
2. **MinHash 签名**：使用 k 个哈希函数，对每个文档的 n-gram 集合独立哈希。取每个哈希函数下的最小值，形成一个 k 维向量作为该文档的"签名"。
3. **Jaccard 相似度**：两个文档的 MinHash 签名相等的概率，恰好等于它们 n-gram 集合的 Jaccard 相似度：

   $$
   P[\text{MinHash}(A) = \text{MinHash}(B)] = \frac{\vert S_A \cap S_B\vert }{\vert S_A \cup S_B\vert } = J(A, B)
   $$

4. **LSH (Locality Sensitive Hashing)**：将 MinHash 签名分成多个 band。如果两个文档在至少一个 band 中签名完全相同，则它们被判定为近似重复。
5. **去重决策**：对每个近似重复的文档对，保留一个代表文档（通常是最长的或最早的版本）。

**参数选择指南**：
- **n-gram 大小**：通常使用 5-gram 或 8-gram。更大的 n-gram 对小的编辑（如插入一个词）更敏感。
- **签名维度 k**：通常 128-256。维度越高，Jaccard 相似度估计越精确，但存储开销增大。
- **band 和 row 的数量**：决定了 Jaccard 相似度的判定阈值。例如，k=128, bands=16, rows=8 → 阈值约 0.8（即约 80% 的 n-gram 重叠时判定为重复）。
- **相似度阈值**：一般设定在 0.5-0.8 之间。较低的阈值会去除更多数据（可能误杀合法变体），较高的阈值可能漏过模板化重复。

**大规模 MinHash 的工程挑战**：
- 万亿 token 级别的语料库有数百万个文档。两两比较的复杂度为 O(N^2)，不可行。
- LSH 将复杂度降低到 O(N)，但需要精心设计 MapReduce/Spark 作业（如 Pile 的去重使用了 Spark + MinHashLSH）。
- 存储 MinHash 签名（每个文档 128 个 32-bit 整数）的总体开销在数 TB 级别被接受范围内。

### 11.4.5 跨数据集去重

一个重要但常被忽视的步骤：当模型使用多个来源的数据时（如 Common Crawl + Wikipedia + Books），需要在**跨来源**的级别上去重。否则，Wikipedia 的文本可能已经出现在 Common Crawl 中（被无数网站转载），导致这些文本被训练两次甚至更多次。

Dolma (Soldaini et al., 2024) 的流水线是一个典例：它包含 Common Crawl 内去重、各来源内去重、以及跨来源去重三个层次。DCLM 的实验表明，跨来源去重带来的提升可与来源内去重相媲美。

### 11.4.6 去重的效果：实证证据

多项研究一致证实了去重对模型性能的积极影响：

- **Lee et al. (2022)**: 在 C4 上去重（MinHash, 阈值 0.8）后，模型在下游任务上的平均性能提升 2-4%，尽管训练 token 数减少了约 10%。更重要的是，模型生成文本的逐字记忆率（verbatim memorization rate）下降了 7-10 倍。
- **RefinedWeb (Penedo et al., 2023)**: Falcon 模型的数据论文指出，严格的去重（URL + 文档级精确 + 近似）是 RefinedWeb 与 C4 之间质量差异的最大单一因素之一。
- **DCLM (Li et al., 2024)**: 在系统消融实验中，去重（尤其是含跨来源去重）是数据流水线中效益最高的环节之一，仅次于质量过滤。
- **Tirumala et al. (2023)**: 发现去重对大规模模型的帮助大于小模型——模型容量越大，越容易"利用"重复数据进行记忆化而非泛化。

**CS 336 的核心信息**：去重是回报最高的数据工程投资之一。它成本可控（在 MapReduce 作业中运行 MinHash），效果显著（2-4% 下游改进），且几乎从不损害性能。如果你只能对数据流水线做一个改进，请选择更好的去重。

## 11.5 数据混合与领域权重

### 11.5.1 问题定义

在收集和清洗多个来源的数据后，面临的下一个问题是如何**混合**它们。不同来源的数据在质量、风格、领域、规模上差异巨大：

| 来源 | 文本可用量 | 质量 | 多样性 |
|------|-----------|------|--------|
| Common Crawl (过滤后) | 数万亿 token | 中等 | 极高 |
| Wikipedia | 数十亿 token | 高 | 中 |
| Books | 数百亿 token | 高 | 低 |
| Code | 数千亿 token | 中 | 中 |
| Academic Papers | 数百亿 token | 高 | 低 |

最简单的混合策略是"按比例混合"（proportional mixing）：每个来源在训练批次中出现的概率与其原始数据量成正比。但这种方法显然有问题——它让模型大量训练在 Common Crawl 的中等质量文本上，而少量见到高质量的 Books 和 Wikipedia。

### 11.5.2 上采样与启发式权重

**上采样（Upsampling）** 是最直接的数据混合策略：人为提高高质量来源的采样概率。

**GPT-3 的数据权重（Brown et al., 2020）**：

| 来源 | 原始数据量 | 训练混合权重 | 有效上采样倍数 |
|------|-----------|-------------|--------------|
| Common Crawl | 60% | 60% | 1x |
| WebText2 | 22% | 22% | 1x |
| Books1 | 8% | 8% | 1x |
| Books2 | 8% | 8% | 1x |
| Wikipedia | 3% | 3% | 1x |

GPT-3 的比例相对保守，基本接近原始数据比例。

**LLaMA 的数据权重（Touvron et al., 2023）**：

| 来源 | 训练混合权重 |
|------|-------------|
| CommonCrawl (过滤后) | 67.0% |
| C4 | 15.0% |
| GitHub | 4.5% |
| Wikipedia | 4.5% |
| Books | 4.5% |
| ArXiv | 2.5% |
| StackExchange | 2.0% |

LLaMA 对 Wikipedia、Books、ArXiv 和 StackExchange 给予了远超其原始比例的权重（按原始比例，Wikipedia 可能只占 < 1%）。

**设计启发式权重的指导原则**：
1. **高质来源上采样**：即使 Wikipedia 只占可用数据的不到 1%，也要给它 3-5% 的训练批次。原因是它的质量密度远高于 Common Crawl。
2. **多样性保留**：不要过度上采样某个来源——如果 50% 的训练数据来自 Wikipedia，模型将无法学习到网络文本的丰富风格和表达方式。
3. **代码和对话的专门权重**：代码数据即使占比不高（如 5-10%），对推理能力的贡献可能远超这个比例暗示的。

### 11.5.3 退火调度（Annealing Schedules）

数据混合不必在整个训练过程中保持不变。一个越来越流行的策略是**退火调度**：在训练的后期阶段（最后 10-30% 的训练步骤）改变数据混合的配比，通常是将更多比例分配给高质量数据。

**动机**：
- 训练前期，模型需要学习广泛的语言模式和知识——多样性高于质量。
- 训练后期，模型接近收敛，此时用最高质量的数据进行"精调"，可以提升最终性能而几乎不增加数据多样性风险。

**具体实践**：
- **LLaMA 2 的风格**：虽然 LLaMA 2 没有明确使用退火，但其训练过程中的固定上采样（如上表）起到了类似效果——在标准余弦衰减的后期，高质量数据的影响因为学习率降低而相对增强。
- **DeepSeek V2**: 在训练的最后阶段提高了高质量数学和代码数据的比例。
- **Gemma 家族**：使用了数据课程的变体，在训练后期增加代码和数学数据的比例。

**退火的注意事项**：
- 退火阶段的数据量相对于总训练量通常很小（10-20%），因此有限的高质量数据（如 Wikipedia）足以覆盖退火需求。
- 退火效果与学习率衰减交互——在低学习率阶段改变数据分布，模型更容易适应新的数据混合，而不会丢弃前期学到的广泛知识。
- 退火的一个极端形式是"预训练 + 微调"——先在大规模混合数据上预训练，再在高质量子集上继续训练几个 epoch。

### 11.5.4 DoReMi：基于优化理论的数据权重学习

Xie et al. (2023) 提出的 **DoReMi (Domain Reweighting with Minimax Optimization)** 是数据混合策略的范式转变：它不依赖启发式或人工判断，而是通过优化理论自动发现最优的领域权重。

**核心原理**：

DoReMi 使用一个小的**代理模型**（proxy model，如 280M 参数）来探究最优的数据混合比例，然后将这个比例应用到大型模型的训练中。

具体过程：
1. 在多个领域的数据上训练一个**参考模型**（使用均匀数据混合）。
2. 在多个领域的数据上训练一个**分布鲁棒优化器**（DRO），它的目标是最大化在所有领域上的最小提升（即提升"最弱领域"的性能——minimax 优化的精髓）。
3. 训练过程中，DRO 优化器动态调整每个领域的采样权重：提升大的领域降权重，提升小的领域增权重。
4. 训练结束后，取得到的领域权重作为"最优混合比例"。
5. 用这个最优比例训练大型的目标模型。

**实证结果**：
- 使用 280M 代理模型找到的权重，应用到 8B 参数模型上，与使用默认权重的模型相比，下游任务平均提升约 6%。
- DoReMi 找到的权重与人工启发式权重显著不同——它更加"反直觉"。例如，某个看起来低质量的来源在 DoReMi 优化后获得了更高的权重，因为它在提升模型的"弱点"方面特别有效。
- 关键是：280M 模型的训练成本相对于 8B 模型微不足道（约 1% 的 FLOPs），因此用代理模型探寻最优权重是计算高效的。

**CS 336 的洞察**：DoReMi 代表了一种计算驱动的、可证明的数据混合方法论。在资源允许的情况下，使用代理模型进行数据消融实验（而非仅凭直觉或历史惯例）是更严谨的做法。这也呼应了 CS 336 的整体哲学：用数据和实验取代猜测。

### 11.5.5 数据课程的自动设计

比静态数据混合更进一步，**数据课程（Data Curriculum）** 在训练的**不同阶段**使用不同的数据混合。这比退火更系统——它不只是在最后阶段切换，而是为整个训练过程设计一个"课程表"。

**理论基础**：
- **课程学习（Bengio et al., 2009）**：从简单样本开始，逐步引入困难样本，有助于模型更快收敛和达到更好的最终性能。
- 对于 LLM 来说，"简单"和"困难"在预训练阶段通常对应"干净/结构化"和"噪声/多样化"的数据。

**实践形式**：
1. **难度递增**：前期以 Wikipedia 和高质量 web text 为主，后期加入 Books 和学术论文（长程依赖和复杂推理更"难"）。
2. **能力递增**：前期以自然语言为主，中期引入代码，后期增加数学和多语言文本。
3. **反向课程**：有趣的是，有些研究（如 Colt, 2024）发现"先 Code 后语言"的反向课程在提升推理能力上更有效——先用代码建立形式推理基础，再用自然语言增强通用性。

**当前状态**：数据课程是一个活跃的研究领域，尚未形成像去重或质量过滤那样的"标准实践"。但方向是清晰的——非均匀的、随时间变化的数据混合在计算效率和最终性能上都优于均匀混合。

### 11.5.6 数据混合的关键启示

1. **启发式上采样是当前实践的主流**：Wikipedia、Books、Code 等高质量来源值得 2-10 倍的上采样权重。
2. **退火和课程学习正在兴起**：训练末期增加高质量数据的比例几乎总是有利的。
3. **DoReMi 证明了自动权重学习的可行性**：用小型代理模型探索权重空间，再迁移到大型模型，是一种计算高效的严谨方法。
4. **没有"万能配方"**：最优的数据混合取决于目标（推理、对话、编码）、可用数据、模型规模和计算预算。DCLM 的系统消融实验方法值得效仿。

## 11.6 合成数据：从 Self-Instruct 到模型蒸馏

### 11.6.1 合成数据的概念与动机

当现实世界的数据不足以满足训练需求时，**合成数据（Synthetic Data）** 提供了一个强大的补充——用模型生成数据来训练模型。

合成数据在 LLM 发展中的角色经历了三个阶段：

1. **对齐阶段（2022）**：Self-Instruct（Wang et al., 2022）使用 LLM 生成指令-响应对，用于指令微调（SFT）。典型代表是 Alpaca——用 GPT-3.5 生成了 52K 条指令，在 LLaMA 7B 上微调后显著提升了指令遵循能力。

2. **预训练补充阶段（2023-2024）**：Phi 系列（Gunasekar et al., 2023; Javaheripi et al., 2024）证明了用 GPT-4 生成高质量的教科书风格数据，可以训练出小参数但能力惊人的模型。Phi-1（1.3B）在 Python 编程方面接近更大的模型；Phi-3（3.8B）在多项基准上达到 7B 模型的水平。

3. **大规模预训练/后训练阶段（2024-2025）**：合成数据开始进入大规模预训练。DeepSeek V3 使用 R1 推理链作为合成训练数据的一部分；Gemini 模型在训练中广泛使用了自己生成的多模态合成数据。

### 11.6.2 Self-Instruct 范式

Self-Instruct (Wang et al., 2022) 是合成数据的基础范式。其工作流程为：

1. **种子任务池**：从 175 个人工编写的"种子指令"开始。
2. **生成新指令**：将 8 条种子指令作为 few-shot 示例，让 LLM 生成更多指令。
3. **分类与去重**：对生成的指令进行分类（生成、分类、问答等），并使用 ROUGE-L 分数去重。
4. **生成响应**：对每条指令，让 LLM 生成对应的输出。
5. **质量过滤**：过滤掉低质量的输入-输出对（如输出过短、输出中包含"我不能"、"作为 AI 模型"等拒绝模板）。
6. **迭代**：将高质量的生成结果回加到任务池，重复步骤 2。

**Self-Instruct 的关键洞见**：
- 175 条种子指令可以"繁殖"出 52K 条高质量的指令-响应对——这是 scaling 的乘法效应。
- 多样性是合成数据质量的关键。Rouge-L 去重和指令分类多样性检查是防止"模式坍缩"的必要步骤。

### 11.6.3 从大模型蒸馏：Textbooks Are All You Need

Phi 系列模型的核心发现是：**数据的教育价值（educational value）比数据量更重要**。与其训练一个模型吸入整个互联网，不如给它提供精心策划的、"教科书质量"的数据。

Phi-1 的训练数据（Gunasekar et al., 2023）：
- 用 GPT-3.5/GPT-4 生成的 Python 教科书和练习集。
- 数据量仅约 7B token，远小于其他模型的预训练数据。
- 但 Phi-1（1.3B）在 HumanEval 上的 pass@1 达到 50.6%，超过了许多 10B+ 的模型。

Phi-3 更进了一步（Javaheripi et al., 2024）：
- 合成数据涵盖数学、编程、逻辑推理和通用知识。
- 使用"种子 - 增强 - 验证"的三环生成流程，确保合成内容正确性和多样性。
- 3.8B 的 Phi-3-mini 在多项基准上达到了 7B 乃至 13B 模型的水平。

**"教科书数据"的核心特征**：
1. **自包含**：每一段文字在自身范围内应该是有意义的（不需要依赖大量外部上下文）。
2. **递进性**：从一个概念到下一个概念有逻辑递进关系。
3. **正确性**：合成的内容经过了验证步骤（如代码执行验证、数学推导验证）。
4. **多样性**：覆盖多种问题类型、解题风格和难度级别。

### 11.6.4 课程学习中的合成数据

合成数据特别适合实现**课程学习**——从简单到困难逐步增加训练数据的复杂度。

一个典型的课程设计：
1. **第一阶段**：基础概念和简单示例（由 LLM 生成的入门级解释）。
2. **第二阶段**：中等复杂度的推理和解题步骤。
3. **第三阶段**：复杂问题和边缘案例。
4. **第四阶段**：多步推理链和自反思（"验证上一步是否正确，如果不对，重新推理"）。

合成数据的优势在于你可以精确控制每个阶段的难度和多样性——这是使用自然语料（Common Crawl 的随机样本）无法做到的。

### 11.6.5 合成数据的局限与风险

**模型坍缩（Model Collapse）**：
Shumailov et al. (2023, 2024) 的研究发现了一个令人担忧的现象：当模型在**自身或同类模型生成的合成数据**上训练时，会发生"模型坍缩"——模型的输出分布逐渐收窄，最终坍缩到极少数的模式，丧失多样性。这类似于近亲繁殖的遗传退化。

**缓解策略**：
- 确保合成数据与真实数据混合训练（而非全部使用合成数据）。
- 使用不同模型、不同 prompt、不同温度参数来生成合成数据（增加多样性）。
- 在生成过程中使用多样化的约束条件（如不同的问题类型、不同的解题风格）。

**幻觉放大**：
如果教师模型（用于生成合成数据的模型）存在幻觉，学生模型不仅会学会这些幻觉，还可能放大它们——因为幻觉在训练中被重复和强化。

**规模谬误**：
合成数据的一个危险是"看起来很多，实际上很单一"——虽然可以轻松生成数百万条合成样本，但如果生成 prompt 和流程缺乏多样性，这些样本的信息增益很低。**质量的多样性 > 表面上的数量**。

### 11.6.6 合成数据的最佳实践

1. **混合使用**：合成数据应该补充而非替代高质量的自然数据。推荐比例：合成数据占 10-30% 的训练 token。
2. **从强模型蒸馏**：用比目标模型更强的模型（如 GPT-4）生成合成数据。用同级别或更弱的模型生成数据的效果递减。
3. **验证环节不可省略**：特别是代码和数学数据，必须通过执行/验证步骤（而非仅依赖模型的"自我评估"）。
4. **多样性监控**：使用嵌入空间的聚类分析监控合成数据的多样性，及时发现模式坍缩的迹象。
5. **规模适当**：合成数据的规模应该与目标匹配。如果你需要 1B 条合成指令，很可能你需要的不是更多数据，而是更高质量和更多样化的数据。

## 11.7 数据污染、基准泄漏与数据文档化

### 11.7.1 数据污染的定义与严重性

**数据污染（Data Contamination / Benchmark Leakage）** 指的是基准测试（benchmark）中的数据出现在了模型的训练集中。这导致基准评估的指标不再反映模型的"真实能力"，而是模型"背题"的能力。

在 2023-2024 年，这成为 LLM 评估中最受关注的问题之一。当 GPT-4、LLaMA 2 和 Gemini 在 HumanEval、MMLU、HellaSwag 等基准上不断刷新纪录时，社区开始质疑这些成绩中有多少来自能力，有多少来自"见过题目"。

### 11.7.2 数据污染的类型

**精确污染（Exact Contamination）**：
基准中的文本（题目、选项、上下文）以完全相同的形式出现在训练数据中。

**近似污染（Near Contamination）**：
基准文本以高度相似但不完全相同的形式出现（如翻译版本、改述版本、仅修改数字的同一道题）。

**基准泄漏（Benchmark Leakage）**：
更隐蔽的形式是——基准的题目可能不直接出现在训练数据中，但基准的答案、解释或分析文章出现在训练数据中，间接泄漏了信息。

### 11.7.3 检测数据污染的方法

**n-gram 重叠检测**：
将基准文本切分为 n-gram（如 13-gram），检查它们是否在训练数据中出现。如果出现多个匹配的连续 n-gram 形成长段匹配，高度怀疑污染。

**GPT-3 论文中的检测方法（Brown et al., 2020）**：
1. 对每个基准样本，在训练数据中搜索重叠的 n-gram。
2. 如果找到匹配，检查匹配长度是否超过阈值。
3. 去除被污染的样本后重新计算基准分数。
4. 比较去除前后的分数差异。

**LLaMA 论文的改进检测（Touvron et al., 2023）**：
使用更长和更多的 n-gram 大小进行交叉验证，并使用 token 级别的精确匹配（而非字符级别的 n-gram），因为 token 级别的匹配更能反映模型实际看到的内容。

**近似污染检测**：
基于嵌入的模糊搜索——将基准文本和训练文本分别编码为嵌入向量，搜索高余弦相似度的对。这比 n-gram 方法能捕获改述和翻译版本的污染。

### 11.7.4 已知的数据污染事件与影响

| 基准 | 污染事件 | 影响 |
|------|---------|------|
| LAMBADA | 广泛出现在 Common Crawl 中 | 早期模型的 LAMBADA 分数可能被高估 |
| HellaSwag | 部分样本出现在 C4 和 WebText 中 | Moderate impact, ~1-2% score inflation |
| MMLU | 题目和答案被转载到考试论坛和问答网站 | 闭卷考试的"真实性"存疑——模型可能见过题目 |
| HumanEval | Python 题目和解答广泛出现在 GitHub 上 | 代码基准的去污几乎不可能——这些题目天然就是"网上的代码" |
| GSM8K | 数学题被转载到 Discord 和论坛 | 数学基准的污染检测比文本基准更难（相同题目可用不同数字） |

**代表性的去污案例**：
- GPT-3 论文首次系统性地进行了去污分析，将 LAMBADA 分数从原始分数修正为去污分数，为社区树立了良好榜样。
- LLaMA 论文对每个评估基准都报告了"可能的污染比例"。
- 但这仍然不够——因为近似污染的检测远非完美。

### 11.7.5 从数据污染到数据文档化

数据污染的深层问题是**缺乏透明性**。如果模型训练数据不公开（如 GPT-4、Claude、Gemini），外部评估者无法独立进行去污检查——他们只能信任供应商的声明。

这推动了**数据文档化（Data Documentation）** 运动，代表性工作是：

**Datasheets for Datasets（Gebru et al., 2018）**：
提出了每个公开数据集应附带的"数据表"标准，包括：
- 数据集的创建动机和目的
- 数据的构成（来源、语言、时间范围）
- 数据收集过程（爬取、标注、清洗）
- 推荐的用途和限制
- 已知的偏差和伦理问题
- 维护和更新的计划

**Data Provenance Initiative（Longpre et al., 2023）**：
一个大规模的数据溯源研究，追踪了 1,800+ 个文本数据集的来源、许可和使用情况。关键发现包括：
- 大多数数据集没有明确的许可证。
- 数据集之间存在广泛的"复制链"（数据集 A 引用数据集 B 的子集，B 又引用 C），但没有追溯机制。
- 许多"热门"数据集存在严重的许可证不一致——标注的许可证与实际内容来源的许可证可能冲突。

### 11.7.6 防止数据污染的实践建议

1. **在数据流水线中建立去污检查**：在每个基准评估之前，用 n-gram 重叠方法检查训练数据与基准的重叠（这是 DCLM 的标准实践）。
2. **使用动态基准测试**：如 BIG-Bench、DynaBench、HumanEval+——这些基准定期更新题目，减少了"背题库"的可能性。
3. **隐私基准（Private Benchmarks）**：使用不公开发布题目的基准（如 HELM 的部分子集），但代价是透明性和可复现性降低。
4. **文档化一切**：无论数据是否公开，都应该记录数据的来源、处理步骤和已知污染情况。如果数据不公开，至少公开数据加工的方法论（如 Meta 对 LLaMA 数据的确切过滤参数进行了描述）。

## 11.8 法律与伦理考量

### 11.8.1 版权问题：数据使用的法律灰色地带

大语言模型训练数据中的版权问题可能是整个 AI 行业面临的最大法律不确定性。

**核心法律问题**：训练 LLM 时使用受版权保护的文本是否构成侵权？

**美国法下的"合理使用"（Fair Use）辩护**：
AI 公司普遍援引合理使用的四大判断标准：
1. **使用的目的和性质**：训练 AI 是否属于"转换性使用"（transformative use）——将文本转化为一种统计模式，而非重新发布文本本身。
2. **受版权保护作品的性质**：事实性作品（如新闻）比创意性作品（如小说）享有更薄弱的保护。
3. **使用的数量和实质性**：训练数据包含整部作品——这是最不利于 AI 公司的因素。
4. **对潜在市场和价值的影响**：AI 生成的文本是否会替代原创作品的市场？这是当前诉讼的核心争议点。

**代表性诉讼**：
- **Authors Guild v. OpenAI/Microsoft (2023)**：包括 George R.R. Martin、John Grisham 在内的多位知名作家起诉，主张他们的作品被用于训练而未经许可。
- **NYT v. OpenAI/Microsoft (2023)**：纽约时报起诉 ChatGPT 的训练数据和生成输出侵犯其版权。
- **Getty Images v. Stability AI**: 图像生成领域的平行案件。

**欧盟的文本与数据挖掘（TDM）例外**：
2019 年《欧盟数字化单一市场版权指令》（CDSM Directive）引入了 TDM 例外条款：研究机构和文化遗产机构在进行科学研究时可以自由进行文本和数据挖掘。但商业用途的 TDM 受限于权利人的明确选择退出（opt-out）。

**实践中的合规策略**：
- **Opt-out 机制**：提供 robots.txt + 专门的 AI 爬虫排除协议（如 `GPTBot` user-agent）。
- **移除请求**：接受并处理数据移除请求（如 OpenAI 的 Data Removal Request 表单）。
- **许可证过滤**：The Stack 代码数据集只包含宽松许可证（MIT, Apache 2.0, BSD）的代码仓库。
- **"灰色区域"承认**：Meta（LLaMA）和 TII（Falcon）等组织在数据论文中明确表示"我们只使用了公开可用的数据"——等于承认了并未获得版权许可，但主张公开可用等同于可以使用。

**当前状态**（截至 2025 年中）：尚未有终审判决就训练数据中的版权问题做出明确裁定。这一法律不确定性将在未来几年持续存在，且其结果可能重塑整个 LLM 行业的商业模式。

### 11.8.2 PII（个人可识别信息）与隐私

训练数据中混杂着大量的个人可识别信息（Personally Identifiable Information, PII），包括：
- 姓名、地址、电话号码、电子邮件
- 社会安全号码（SSN）、信用卡号码
- IP 地址、用户名
- 身份证号码（不同国家不同格式）

**PII 的来源**：个人主页、论坛帖子、简历网站、公开记录、违规数据泄露的转载……

**PII 的风险**：
1. **隐私泄漏**：模型可能记住并复述训练数据中的个人敏感信息。Carlini et al. (2021) 证明，GPT-2 可以通过精心构造的 prompt 提取训练数据，包括姓名、电话号码和电子邮件。
2. **社会工程攻击**：模型生成的文本中可能无意包含真实的个人信息。
3. **合规风险**：GDPR（欧洲）、CCPA（加利福尼亚）、PIPL（中国）等隐私法规对个人数据的收集和处理有严格规定——即使数据是"公开可用"的。

**PII 检测与脱敏方法**：

1. **正则表达式匹配**：针对不同国家的标准格式（美国的 SSN 格式 `\\d{3}-\\d{2}-\\d{4}`、邮件格式 `...@...`、电话格式等）。这是第一道防线，但 F1 分数有限。
2. **基于 NER 的检测**：使用命名实体识别模型（如 spaCy、Microsoft Presidio）检测 PERSON、EMAIL、PHONE、LOCATION 等实体类型。但 NER 在嘈杂的网络文本上准确率不高。
3. **基于分类器的检测**：训练一个专门的 PII 分类器（如 BigCode 项目中用于 PII 检测的模型），在文档级别判断是否包含 PII。
4. **脱敏 vs 移除**：检测到的 PII 可以被替换为占位符（脱敏，如 `[EMAIL]`、`[PHONE]`），也可以整篇文档被丢弃（移除）。实践中，由于脱敏的复杂性（需要正确识别和替换所有 PII 而不遗漏），许多流水线选择"包含 PII 的文档直接丢弃"这一保守策略。

**PII 过滤的困境**：
- 过度过滤会丢失那些"恰巧包含人名"的研究论文和新闻文章。
- 过滤不足会留下隐私泄漏的风险。
- 对于非英语和非拉丁文字，正则表达式和 NER 的效果都会大幅下降。

### 11.8.3 同意与数据伦理

**"公开可获取"不等于"同意使用"**：
大多数训练数据是从公开网络抓取的。但内容创建者（包括个人博客作者、论坛参与者、开源开发者）可能并未意识到——或未同意——他们的内容被用于训练商业 AI 产品。

**创作者权利运动**：
- "Have I Been Trained?" 网站允许创作者检查自己的作品是否出现在训练数据集中。
- DeviantArt、Stack Overflow 等平台允许用户选择退出 AI 训练。
- GitHub Copilot 引发了开源社区关于"代码是否可以在未经许可的情况下用于训练"的激烈辩论。

**伦理框架的演进**：
学术社区正在从"只要合法就可以"（compliance-based）向"负责任的数据使用"（responsibility-based）框架转变。
- **数据管理（Data Stewardship）**：将训练数据视为需要"管理"的公共资源，而非可以随意提取的"免费原材料"。
- **数据正义（Data Justice）**：考虑数据使用对内容创作者和弱势社区的不成比例影响。
- **数据关怀（Data Care）**：强调数据收集者和使用者对数据主体的关怀义务，类似于医疗领域的关怀伦理。

### 11.8.4 实践中的法律与伦理防护体系

一个负责任的数据流水线应至少包含以下防护措施：

1. **PII 检测与移除**：在文本清洗阶段进行（至少使用正则表达式和 NER 的两层检测）。
2. **毒性内容过滤**：移除仇恨言论、极端暴力和 CSAM 相关内容。
3. **许可证合规检查**：对于代码数据（如 The Stack），根据许可证过滤。
4. **Opt-out 响应机制**：遵守 robots.txt、AI 爬虫协议和个别移除请求。
5. **数据文档化**：发布清晰的数据报告（参考 Datasheets for Datasets 模板），说明数据的来源、处理方法和已知限制。
6. **影响评估**：在发布模型前进行伦理影响评估，特别是针对高风险应用场景（医疗、法律、教育）。

## 11.9 深度推理：数据质量、数据量与模型性能的三角关系

本章的最后，回到开篇提出的核心张力——**数据质量 vs 数据数量**——并给出基于理论推理和实证证据的深度分析。

### 11.9.1 Chinchilla 最优与"质量修正"

Chinchilla 定律（Hoffmann et al., 2022）是 LLM 训练的基石发现之一：在给定的计算预算下，存在一个最优的模型参数量 $$N_{\text{opt}}$$ 和训练 token 数 $$D_{\text{opt}}$$，使得下游性能最大化。具体而言 $$D_{\text{opt}} \approx 20 N_{\text{opt}}$$ ——即每个参数需要约 20 个训练 token。

但 Chinchilla 的实验（以及大多数后续验证）使用的是**质量均匀的数据**（MassiveText 数据集，经过一致的过滤）。在真实世界中，数据是异质的——Common Crawl 的"平均质量"远低于 Wikipedia。

这就引出了一个根本性的修正：**当数据质量提升时，Chinchilla-optimal 的数据量可以降低**。更具体地说：

1. **单位质量的 token**：一个 Wikipedia token 包含的"有效信息"可能相当于 10 个 Common Crawl token。
2. **质量-数量的替代关系**：如果数据质量提高了 $$k$$ 倍（按信息密度衡量），Chinchilla-optimal 的 $$D_{\text{opt}}$$ 可以降低约 $$k$$ 倍。
3. **上限效应**：当数据量达到某个阈值后（对于当前规模的模型，约 1-2T 高质量 token），额外的数据带来的收益递减。此时，提升数据质量比增加数据量更有意义。

**实证支持**：
- Phi 系列（Gunasekar et al., 2023）：用 7B 高质量合成 token 训练的 1.3B 模型，性能超过了用数万亿 Common Crawl token 训练的更大模型。
- FineWeb-Edu：在数据量固定的情况下，更高的教育质量过滤阈值一致地提升了下游性能——即使这意味着丢弃更多的"普通"文本。
- DCLM 的系统消融：在相同 token 数下，质量过滤策略的差异导致了高达 10% 的下游性能差异。

### 11.9.2 为什么去重提升性能——信息论的视角

去重提升性能可以通过信息论给出优雅的解释。

**有效信息量（Effective Information Content）**：

设训练语料 $$\mathcal{D}$$ 包含 $$N$$ 个 token。在完全去重后，剩余 $$N' < N$$ 个唯一 token。但每个 token 携带的信息密度是多少？

如果一段 1,000 token 的文字在语料中出现了 100 次，从模型学习的角度看：
- **无去重**：模型见到这段文字 100 次。第一次提供了所有信息（词汇、句法、事实），后 99 次提供的**新信息接近于零**。模型学会了这段文字的"表面形式"（memorization），但未提取与上下文无关的深层模式。
- **去重**：模型只见到这段文字 1 次，但有更多容量和训练步骤去学习其他 99 段不同的文字。多样性增加了。

**信息密度公式**：

定义语料库的信息密度 $$\rho$$ 为：

$$
\rho(\mathcal{D}) = \frac{I(\mathcal{D}; \theta^*)}{\vert \mathcal{D}\vert }
$$

其中 $$I(\mathcal{D}; \theta^*)$$ 是语料与最优模型参数之间的互信息（即语料对模型能力的贡献）。去重通过移除冗余来降低分母 $$\vert \mathcal{D}\vert $$，同时几乎不降低分子 $$I(\mathcal{D}; \theta^*)$$，从而提高了信息密度 $$\rho$$。

**关键洞察**：Chinchilla 定律中的 $$D_{\text{opt}}$$ 应该被理解为**有效 token 数**，而非原始 token 数。如果 30% 的 token 是重复的，那么实际的 $$D_{\text{opt}}$$ 需要上调约 43% 以补偿信息密度的下降。去重通过移除这些浪费的 token，使训练更高效。

### 11.9.3 数据多样性与质量的权衡

除了质量和数量的二元对立，还有第三个维度：**多样性（Diversity）**。

| 数据来源 | 质量 | 多样性 |
|---------|------|--------|
| Wikipedia | 高 | 中（主题范围有限，NPOV 风格统一） |
| Books | 高 | 低-中（受限于出版商选择和文学风格） |
| Common Crawl (过滤后) | 中 | 高（覆盖几乎所有人类写作主题） |
| 合成数据（GPT-4 生成） | 高 | 低-中（受限于教师模型的分布） |

**质量-多样性权衡（Quality-Diversity Tradeoff）**：

过度追求质量可能导致"多样性坍缩"。如果你只训练 Wikipedia + Books，模型会对网络文本的非正式语体、社交媒体互动、小众领域知识一无所知。这正是为什么即使 Wikipedia 的质量远高于 Common Crawl，模型仍然需要大量 Common Crawl 数据。

**信息覆盖范围 vs 信息精度的权衡**：
- Common Crawl 提供广度（覆盖罕见实体、小众话题、时新事件）。
- Wikipedia/Books 提供深度（可靠事实、复杂推理、文雅表达）。
- 一个良好的数据混合需要同时满足两者。

**CS 336 的洞察**：不要试图将所有数据提升到同一"质量标准"。不同的数据来源有不同的生态位——Common Crawl 的价值在于其**多样性**（覆盖了 W Wikipedia 永远不会触及的角落），Wikipedia 的价值在于其**准确性**。数据工程的艺术在于让它们各司其职，而非将 Common Crawl 过滤到"接近 Wikipedia"后丢弃那部分"不够 Wikipedia 但足够多样"的文本。

### 11.9.4 未来的方向

数据工程正在从"手艺活"（manual ad-hoc rules）向"系统工程"（principled optimization）转变。几个关键趋势：

1. **自动化数据流水线设计**：DCLM 的系统消融和 DoReMi 的权重学习是早期代表。未来的流水线将更多依赖在代理模型上的自动搜索，而非人工规则。
2. **数据质量的形式化度量**：取代当前的"好坏"二分，建立连续的质量度量（如教育价值分数、信息密度分数），使质量过滤成为一个可微调而非硬判定的问题。
3. **合成数据的可控化**：更精细的课程设计、更可靠的多样性保证、更系统的模型坍缩防御。
4. **数据溯源与治理**：随着法律和伦理压力的增加，数据溯源（每个训练样本的来源、许可、处理历史）将从"可选的好习惯"变为"必要的合规要求"。

## 11.10 本章小结

数据是大语言模型的根基。架构决定了能力的天花板，而数据决定了模型能够多接近这个天花板。本章从全链路视角系统梳理了 LLM 数据工程的核心实践与深层原理：

1. **数据来源**：Common Crawl 是规模之锚，Wikipedia、Books、Code、学术论文各自提供了不可替代的信号。不同来源的互补性决定了数据多样性对模型性能的关键贡献。

2. **Web 数据处理流水线**：从 HTML 提取到样板移除、语言检测、质量过滤（启发式/困惑度/分类器），每个环节都在"保留多样性"与"提升质量"之间寻找平衡。过滤过少则噪声过多，过滤过多则多样性丧失。

3. **数据去重**：精确去重（哈希）、近似去重（MinHash + LSH）、URL 去重和跨来源去重是多层防御。去重是回报最高的单一数据工程操作——提升泛化、减少记忆化、提高信息密度。

4. **数据混合与权重**：启发式上采样是当前实践主流；退火调度和学习率调度在训练末期可以用高质量数据精调模型；DoReMi 代表了自动化权重学习的未来方向。

5. **合成数据**：Self-Instruct 和模型蒸馏（Phi 系列）展示了质量优先于数量的可能。但模型坍缩和幻觉放大是合成数据的核心风险——多样性监控和真伪混合是必要的防御。

6. **数据污染**：基准泄漏破坏了 LLM 评估的可靠性。n-gram 重叠检测是基础手段，但近似污染仍需更好的检测方法。数据文档化和透明度是长期解方。

7. **法律与伦理**：版权争议悬而未决；PII 检测与移除是技术和合规的双重挑战；"公开可获取"不等于"同意使用"——伦理框架正在从合规导向向责任导向演进。

8. **深度原理**：数据质量修正了 Chinchilla 最优的 $$D_{\text{opt}}$$——高质量 token 的信息密度可能是低质量 token 的 10 倍。去重通过提高信息密度来改善训练效率。质量与多样性的权衡决定了最优数据混合的复杂性。

**CS 336 的核心信息**：数据工程不是一次性的预处理，而是一个需要与模型训练一起迭代优化的闭环系统。最好的数据配方不是猜出来的，而是像 DCLM 那样——在每个数据决策上训练小型消融模型，用实证证据替代直觉。
