> 原课程: CS 336 Language Modeling from Scratch, Stanford, Spring 2025
> 主讲: Tatsu Hashimoto
> 主题: Parallelism for LLM Training — Data, Model, and Pipeline Parallelism
> 中译稿 — 从零实现视角的系统并行扩展版

## 一、本章概要

**核心问题**: 第 5 章和第 6 章教会了我们如何在单个 GPU 上高效执行 Transformer 的计算——通过理解硬件架构、算子融合和分块策略。但现实是，现代 LLM 的参数规模已经远远超出了单 GPU 的物理极限: GPT-3 175B 需要约 350 GB 的 fp16 参数存储，而一块 H100 只有 80 GB 显存。如何将模型和数据拆分到数十、数百甚至数千个 GPU 上，同时保持接近线性的加速比——这是**分布式训练**要回答的核心问题。

**动机**: 分布式训练不仅仅是"用更多 GPU 跑得更快"。它是一个关于**通信与计算的权衡**的系统设计问题: 不同的并行策略有不同的通信模式、通信量和内存节省倍数，而你的 GPU 集群拓扑结构(节点内 NVLink 带宽 vs. 节点间 InfiniBand/RoCE 带宽)决定了你应该在哪里使用哪种并行策略。理解这些权衡是设计可扩展训练系统的前提。

**本章主线**: 本章从 CS 336 "从零实现"的视角出发，系统性地覆盖五种并行范式:

1. **数据并行 (Data Parallelism)** — 包括朴素 DDP 和 ZeRO 阶段 1/2/3(FSDP)，核心洞察: 通信量在所有阶段都是 $$2 \times \#\text{params}$$(带宽受限时), 但内存节省随阶段递增。
2. **流水线并行 (Pipeline Parallelism)** — 按层切分模型，通过微批次(micro-batch)隐藏流水线气泡, 适合跨节点慢速链路。
3. **张量并行 (Tensor Parallelism)** — 按矩阵的列/行维切分权重, 通信量大但无气泡, 适合节点内高速链路。
4. **序列并行 (Sequence Parallelism)** — 沿序列维切分 LayerNorm/Dropout 等逐点操作, 使激活内存真正线性缩放。
5. **专家并行 (Expert Parallelism)** — MoE 模型中按专家切分, 通过 token 路由 all-to-all 通信替代矩阵切分。

学完本章后，你应当能够:

1. 从通信量($$2 \times \#\text{params}$$ 的 all-reduce)、内存(每 GPU 需全量参数副本)和批量大小三个维度分析朴素 DDP 的局限性。
2. 解释 ZeRO 三个阶段的核心差异: 为何阶段 1 和 2 通信量相同(均为 $$2 \times \#\text{params}$$), 但阶段 3 需要额外的 all-gather(总通信量 $$3 \times \#\text{params}$$)?
3. 推导流水线气泡比例公式 $$\frac{n_\text{stages} - 1}{n_\text{micro}}$$, 并解释零气泡(zero-bubble)流水线如何通过拆分反向传播来消除气泡。
4. 区分行张量并行和列张量并行的通信模式: 为什么列并行的前向通信是 all-reduce 而反向是 identity? 行并行恰好相反?
5. 在给定的集群拓扑(如 8 节点、每节点 8 GPU)上, 为特定规模的模型设计 TP/PP/DP 的最优组合策略。
6. 阅读并理解真实大规模训练系统的并行配置(DeepSeek-V3, Llama 3 405B, Qwen 3 等)。

> 核心主题: 分布式训练的本质是在**通信带宽**与**内存容量**之间做权衡——不同的并行策略代表了这个权衡空间中的不同帕累托最优点。

## 二、单 GPU 的极限与多 GPU 并行的必要性

### 2.1 算力极限

全球最快的超级计算机拥有 ExaFLOPs 级别的算力(1 ExaFLOP/s = $$10^{18}$$ 次浮点运算/秒)。而一块 H100 的 bf16 峰值算力约为 989 TFLOPs/s($$9.89 \times 10^{14}$$)。即便不考虑通信损失，训练一个万亿参数模型也需要远超单 GPU 的算力。

但算力并非唯一瓶颈——甚至不是最紧迫的瓶颈。

### 2.2 显存: 真正的硬约束

考虑训练时单 GPU 上需要驻留的显存构成(以 bf16 混合精度训练为例, 每参数字节数):

| 组件 | 精度 | 每参数字节数 |
|:---|:---|:---:|
| 模型参数 | bf16/fp16 | 2 |
| 梯度 | bf16/fp16 | 2 |
| 优化器主权重 (master weights) | fp32 | 4 |
| Adam 一阶矩估计 ($$m_t$$) | fp32 | 4 |
| Adam 二阶矩估计 ($$v_t$$) | fp32 | 4 |
| **合计 (不含激活)** | | **16 字节/参数** |

这意味着训练时需要 **16 字节/参数** 的静态显存。一块 80 GB H100 最多容纳:

$$
\frac{80 \times 10^9}{16} = 5\text{B params (纯静态参数)}
$$

而实际训练中还有激活内存(activations)、KV Cache、通信缓冲区等开销。因此, 单 GPU 通常只能训练 1-2B 参数以下的模型——GPT-3(175B)、Llama 3(405B)、DeepSeek-V3(671B MoE)等模型远远超出了单 GPU 的物理限制。

### 2.3 解决方案: 多 GPU 多节点并行

核心思路: 将显存和计算需求拆分到多个 GPU 和节点上。这引出两个理想目标:

- **线性显存缩放**: 模型可训练的最大参数量随 GPU 数量线性增长。
- **线性计算缩放**: 模型 FLOPs 随 GPU 数量线性增长(即每个 GPU 的计算量保持不变)。

现实与理想的差距在于**通信开销**——不同并行策略引入不同的通信模式, 我们需要理解这些通信模式以做出正确的设计决策。

### 2.4 节点内 vs 节点间通信

GPU 集群的通信分为两个层次:

- **节点内 (Intra-node)**: GPU 通过高速互联通信, 如 NVLink 4.0(H100: 900 GB/s 双向, 每 GPU 对 GPU 450 GB/s) 或 NVSwitch(全互联, 所有 GPU 之间全带宽通信)。
- **节点间 (Inter-node)**: 节点通过 InfiniBand(NDR: 400 GB/s 每端口)或 RoCE 网络通信, 带宽远低于节点内, 延迟也更高。

**经验法则**: 在 8 卡 H100 节点内, NVLink 提供 450 GB/s 的 GPU 对 GPU 带宽, 而跨节点的 InfiniBand 通常只有约 50 GB/s(一个数量级的差异)。这意味着**高通信量的并行策略(如张量并行)应优先在节点内使用**, 而**低通信量的策略(如流水线并行)可用于跨节点**。

## 三、集合通信原语

在深入具体并行策略之前, 必须先理解多 GPU 通信的基本原语。分布式训练中的通信可以用四种集合操作描述:

| 操作 | 语义 | 通信量 (每 GPU) | 图示 |
|:---|:---|:---:|:---|
| **Broadcast** | 一个 GPU 的数据复制到所有 GPU | $$1 \times \text{data}$$ | 一对多 |
| **Reduce** | 所有 GPU 的数据按元素归约(如求和)到一个 GPU | $$1 \times \text{data}$$ | 多对一 |
| **All-Gather** | 每个 GPU 持有一部分数据, 收集后所有 GPU 都获得完整拼接 | $$(N-1) \times \text{data}_\text{per GPU}$$ | 多对多 |
| **Reduce-Scatter** | All-Gather 的对偶: 归约后将结果分散到各 GPU | $$(N-1) \times \text{data}_\text{per GPU}$$ | 多对多(分散) |
| **All-Reduce** | 归约 + Broadcast: 所有 GPU 获得相同的归约结果 | $$2 \times \frac{N-1}{N} \times \text{data} \approx 2 \times \text{data}$$ | 多对多 |

### 3.1 All-Reduce 的分解: Reduce-Scatter + All-Gather

一个关键的实现细节是: **all-reduce 可以分解为 reduce-scatter 后接 all-gather**:

```
All-Reduce(梯度和) = Reduce-Scatter(梯度分散归约) + All-Gather(参数收集)
```

这是因为在带宽受限(即通信链路带宽饱和)时, 这种两步方法达到了理论最优的通信量: $$2 \times \frac{N-1}{N} \times \#\text{params} \approx 2 \times \#\text{params}$$。每个 GPU 发送和接收各约 $$\#\text{params}$$ 的数据量。

ZeRO 系列优化器的核心洞察正是利用了这个分解: 将 reduce-scatter 和 all-gather 分别映射到优化器状态的分片管理上。

### 3.2 TPU vs GPU: 通信拓扑的差异

不同硬件平台的通信拓扑对其适合的并行策略有重要影响:

**TPU 网络 (传统)**:
- 环形网格 (toroidal mesh): 每个 TPU 与其四个邻居直接相连
- 优势: 对于结构化的张量并行通信(all-reduce)非常高效
- 局限: 非结构化通信(如 MoE 的 all-to-all)效率较低

**GPU 网络**:
- 全互联(up to 256 GPUs via NVSwitch): 任意 GPU 之间直接通信
- 优势: 对任意通信模式(包括 MoE 的 all-to-all)都很高效
- 代价: 全互联的硬件成本和功耗更高

**最新趋势**: TPU8i/t 引入了更接近树形拓扑的网络(适合 MoE), 而 Virgo(TPU 的 scale-out 网络)采用了交换网络。这表明硬件设计正在向**更灵活的非结构化通信**方向演进, 以更好地支持 MoE 等新架构。

## 四、数据并行: 从朴素 DDP 到 ZeRO-3/FSDP

### 4.1 朴素数据并行 (Naive DDP)

**原理**: 将批次大小为 $$B$$ 的数据均匀拆分到 $$M$$ 个 GPU 上, 每个 GPU 在自己的数据子集上计算梯度, 然后通过 all-reduce 同步梯度。

**SGD 视角**:

$$
\theta_{t+1} = \theta_t - \eta \sum_{i=1}^{B} \nabla f(x_i)
$$

在数据并行中, 每个 GPU 计算 $$\sum_{i=1}^{B/M} \nabla f(x_i)$$(局部梯度), 然后 all-reduce 求和。

**朴素 DDP 在三个维度上的表现**:

| 维度 | 表现 | 评价 |
|:---|:---|:---|
| 计算缩放 | 每个 GPU 处理 $$B/M$$ 个样本 | 线性缩放, 但需要足够大的批大小 |
| 通信开销 | 每步 all-reduce 梯度, 通信 $$2 \times \#\text{params}$$ | 在大批大小下可接受; 小批大小下成为瓶颈 |
| 显存缩放 | **无**——每个 GPU 需要完整的参数副本 | **致命缺陷** |

### 4.2 朴素 DDP 的显存危机

回顾第 2.2 节的显存构成: 每个 GPU 需要存储完整的参数、梯度、优化器主权重和 Adam 状态。对于 fp16/bf16 混合精度训练:

- 2 bytes: fp16/bf16 模型参数
- 2 bytes: fp16/bf16 梯度
- 4 bytes: fp32 主权重 (SGD 中用于累积更新)
- 4 bytes: fp32 Adam 一阶矩 (optimizer state)
- 4 bytes: fp32 Adam 二阶矩 (optimizer state)

**合计 16 字节/参数**——而且每个 GPU 都需要完整的一份。这意味着朴素 DDP **在每个 GPU 上复制了整个训练状态**, 显存效率极低。

### 4.3 ZeRO (Zero Redundancy Optimizer) 总览

ZeRO 的核心思想: **将昂贵的训练状态分片(shard)到多个 GPU 上**——利用 reduce-scatter + all-gather 等价性, 将通信成本隐藏在计算中。

**核心洞察**: 既然 all-reduce 可以分解为 reduce-scatter + all-gather, 为什么不在 reduce-scatter 后让每个 GPU **只更新自己负责的那部分参数**(使用对应的 optimizer state 分片), 然后 all-gather 收集更新后的参数?

ZeRO 的三个阶段代表了分片的激进程度:

| 阶段 | 分片内容 | 通信量 | 每 GPU 显存 (bytes/param) |
|:---|:---|:---:|:---:|
| 朴素 DDP | 无分片 | $$2 \times \#\text{params}$$ | 16 |
| ZeRO-1 | Optimizer state | $$2 \times \#\text{params}$$ | $$2 + 2 + \frac{12}{N_\text{GPU}}$$ |
| ZeRO-2 | Optimizer state + 梯度 | $$2 \times \#\text{params}$$ | $$2 + \frac{14}{N_\text{GPU}}$$ |
| ZeRO-3 (FSDP) | 一切(包括参数) | $$3 \times \#\text{params}$$ | $$\frac{16}{N_\text{GPU}}$$ |

### 4.4 ZeRO-1: Optimizer State 分片

**高层思路**: 将 optimizer state(Adam 的一阶和二阶矩)拆分到不同 GPU 上。每个 GPU 仍然持有完整的参数和梯度。

**执行流程**:

1. **前向 + 反向**: 每个 GPU 在其数据子集上计算完整梯度(每个 GPU 拥有完整参数副本)
2. **Reduce-Scatter 梯度**: 将梯度归约后分散到各 GPU, 通信 $$1 \times \#\text{params}$$
3. **局部更新**: 每个 GPU 使用自己的 optimizer state 分片更新其负责的参数子集
4. **All-Gather 参数**: 收集更新后的参数到所有 GPU, 通信 $$1 \times \#\text{params}$$

**结论**: ZeRO-1 在通信量与朴素 DDP 完全相同($$2 \times \#\text{params}$$)的前提下, 将 optimizer state 显存减少了 $$N_\text{GPU}$$ 倍。在带宽受限时, **ZeRO-1 是免费的**——没有任何通信代价增加。

### 4.5 ZeRO-2: 梯度也分片

**高层思路**: 受 ZeRO-1 成功的鼓舞, 进一步将梯度也分片存储。关键挑战: 每个 GPU 仍需计算完整梯度(因为数据是并行的), 但不能随时物化完整梯度向量。

**执行流程**:

1. **反向传播 + 增量通信**: 
   - 每计算完一层的梯度, 立即 reduce-scatter 将该层梯度发送到负责该层的 GPU
   - 一旦该层梯度在后续反向图中不再需要, 立即释放显存
2. **局部更新**: 每个 GPU 使用自己的 optimizer state 分片更新其负责的参数
3. **All-Gather 参数**: 收集更新后的参数

**关键实现细节**: ZeRO-2 要求在反向传播过程中**增量式地**进行通信和释放, 以保持较低的峰值显存。这需要框架级别的支持(PyTorch 的 FSDP 或 DeepSpeed)。

### 4.6 ZeRO-3 / FSDP: 参数也分片

**高层思路**: 将一切——参数、梯度、optimizer state——全部分片。每个 GPU 在计算某层时才临时 gather 该层的完整参数。

**执行流程**:

1. **All-Gather 参数**(前向传播前): 每个 GPU 收集当前层需要的完整参数, 通信 $$1 \times \#\text{params}$$
2. **前向传播**: 使用完整参数计算, 完成后立即释放非本地分片的参数副本
3. **All-Gather 参数**(反向传播前): 再次收集完整参数(如果未缓存), 通信 $$1 \times \#\text{params}$$
4. **反向传播**: 计算梯度, 立即 reduce-scatter 发送到负责的 GPU, 通信 $$1 \times \#\text{params}$$
5. **局部更新**: 每个 GPU 更新自己管理的参数分片

**总通信量**: $$3 \times \#\text{params}$$, 比朴素 DDP 多 50%。但显存节省为 $$N_\text{GPU}$$ 倍——这是一个合理的权衡。

### 4.7 FSDP 的关键优化: 通信与计算重叠

FSDP 并非简单的"先 gather 再计算"。其实际实现包含两个重要优化:

**优化 1: 增量计算/增量通信**

在任何时刻, 只有当前正在计算的层需要完整的参数。当一个层的计算完成后, 其 gather 的参数副本立即释放。对于 Transformer 模型, 这意味着:

- 在处理第 $$L$$ 层时, 第 $$L-1$$ 层的参数已经被释放
- 峰值显存 ≈ 最大层的参数 × 2(gather 副本 + 本地分片) + 激活内存

**优化 2: 重叠通信与计算**

以 MLP 中的两个矩阵乘法 $$(W_1, W_0) x = y$$ 为例:

```
时间轴:
|---- All-Gather W0 ----|---- All-Gather W1 ----|
|---- 计算 W0 @ x -------|---- 计算 W1 @ ... ----|
```

通过在前向传播计算 $$W_0 x$$ 的同时异步发起对 $$W_1$$ 的 all-gather, 通信延迟被计算完全隐藏。

### 4.8 ZeRO 实战: 8 卡 A100 80G 可训练多大规模?

假设使用纯 bf16 训练(含 bf16 Kahan 求和的 master weight), 每参数 12 字节:

| 方法 | 最大参数量 (8×A100 80G) | 每参数字节数公式 |
|:---|:---:|:---|
| 基线 (朴素 DDP) | 6.67 B | 12 |
| ZeRO-1 | 16 B | $$2 + 2 + 4/N + 4/N$$ (bf16 Kahan) |
| ZeRO-2 | 24.62 B | $$2 + (2 + 4 + 4)/N$$ |
| ZeRO-3 (FSDP) | 53.33 B | $$12/N$$ |

**结论**: 8 张 A100 可以训练一个约 53B 参数的稠密模型(使用 ZeRO-3), 但需要谨慎管理激活内存。

### 4.9 数据并行的剩余问题

尽管 ZeRO 解决了显存问题, 数据并行仍有两个根本局限:

1. **批量大小限制**: 数据并行的并行度 $$\leq$$ 全局批大小。当 GPU 数量接近批大小时, 每个 GPU 只处理几个样本, 通信开销占比急剧上升。
2. **模型不匹配**: ZeRO-1/2 **不减少激活内存**; ZeRO-3 减少参数/梯度/状态显存但**不减少激活内存**。对于长上下文训练, 激活内存(尤其是 attention 的中间矩阵)可能是最大的显存消耗者。

**因此, 当数据并行走到极限时, 必须在模型本身上做拆分——这就是模型并行。**

## 五、流水线并行

### 5.1 朴素层并行及其问题

最直观的模型拆分方式: 将 Transformer 的层按序分配给不同 GPU。

```
GPU 0: Embedding → Layer 0-3
GPU 1: Layer 4-7
GPU 2: Layer 8-11
GPU 3: Layer 12-15 → LM Head
```

**问题**: 在反向传播完成之前, 每个 GPU 大部分时间在**空转**。假设有 $$n$$ 个 GPU:

$$
\text{每个 GPU 的活跃时间占比} = \frac{1}{n}
$$

以 4 个 GPU 为例, 每个 GPU 只有 25% 的时间在计算——其余 75% 的时间在等待前一个 GPU 完成前向或后一个 GPU 完成反向。

### 5.2 微批次流水线: GPipe 方案

**解决方案**: 将批次拆分为更小的**微批次 (micro-batches)**, 让它们像流水线一样依次流过各 GPU:

```
时间 →
GPU 0: [F0] [F1] [F2] [F3] [B0] [B1] [B2] [B3]
GPU 1:      [F0] [F1] [F2] [F3] [B0] [B1] [B2] [B3]
GPU 2:           [F0] [F1] [F2] [F3] [B0] [B1] [B2] [B3]
GPU 3:                [F0] [F1] [F2] [F3] [B0] [B1] [B2] [B3]
```

其中 `F` 表示微批次的前向传播, `B` 表示反向传播。前向与反向之间的空隙称为**流水线气泡 (pipeline bubble)**。

**气泡比例**:

$$
\text{Bubble Ratio} = \frac{n_\text{stages} - 1}{n_\text{micro}}
$$

当微批次数量远大于流水线阶段数时($$n_\text{micro} \gg n_\text{stages}$$), 气泡比例趋近于零。但这也意味着需要较大的全局批大小——而我们在第 2 章已经讨论过, 批大小有递减收益。

### 5.3 为什么仍使用流水线并行?

流水线并行看起来效率不高, 但有两个重要优势:

1. **显存节省**: 每个 GPU 只存储 $$\frac{1}{\text{PP}}$$ 的模型层, 这比数据并行(每个 GPU 全量参数)好得多。
2. **通信特性优异**: 流水线并行只需要在阶段边界传递**激活张量**(形状 $$b \times s \times h$$, 即 batch × sequence × hidden), 这是**点对点**通信(只需发给下一个 GPU), 而非 all-reduce 那样的全带宽通信。通信量为 $$b s h$$ 每微批次——远小于张量并行的通信量。

**因此, 流水线并行通常用在慢速网络链路上(如跨节点), 作为一种以低通信代价获得显存缩放的方式。**

### 5.4 1F1B (One-Forward One-Backward) 调度

GPipe 的标准调度是"先完成所有微批次的前向, 再完成所有反向"。这导致大量的激活需要缓存(因为反向需要中间激活来计算梯度)。**1F1B 调度**改善了内存使用:

```
时间 →
GPU 0: [F0] [F1] [F2] [F3] [B0] [F4] [B1] [F5] [B2] [B3]
GPU 1:      [F0] [F1] [F2] [F3] [B0] [F4] [B1] [F5] [B2] [B3]
...
```

1F1B 的核心思想: 在完成足够多的前向微批次以启动反向后(称为 warm-up), 交替执行前向和反向。这减少了同时需要驻留的激活数量。

### 5.5 零气泡流水线 (Zero-Bubble Pipeline)

更激进的优化: 将反向传播拆分为两部分:

1. **反向传播激活梯度 (B)**: 对输入和激活的反向传播(输入梯度)
2. **计算权重梯度 (W)**: 对本层权重的反向传播(权重梯度)

关键洞察: **权重梯度的计算 (W) 可以在任何时候进行**——只要激活梯度已经计算出来了, 权重梯度的计算不依赖于其他层的状态。

通过将反向拆分为 B 和 W 两个阶段, 并将 W 填充到气泡中, 可以进一步降低甚至消除流水线气泡——代价是需要更复杂的调度逻辑和额外的通信带宽。

### 5.6 流水线并行的性能对批大小高度敏感

流水线并行的气泡比例 $$\frac{n_\text{stages} - 1}{n_\text{micro}}$$ 决定了: **批大小(微批次数量)是隐藏气泡的关键**。当微批次数量不足时, 流水线的利用率急剧下降。

在批大小受限的场景(如微调、RLHF)中, 流水线并行可能不是最佳选择——此时张量并行(无气泡)或 FSDP(ZeRO-3)更合适。

## 六、张量并行

### 6.1 沿宽度切分模型

流水线并行是沿**深度**维(层数)切分模型。另一种思路是沿**宽度**维(每层的矩阵维度)切分。

**数学基础**: 矩阵乘法的分块分解:

$$
Y = X A = X [A_1, A_2] = [X A_1, X A_2]
$$

其中 $$A_1, A_2$$ 是将 $$A$$ 按**列**切分的子矩阵。两个部分积可以分别在不同 GPU 上计算, 最后拼接。

$$
Z = Y B = [Y_1, Y_2] \begin{bmatrix} B_1 \\ B_2 \end{bmatrix} = Y_1 B_1 + Y_2 B_2
$$

其中 $$B_1, B_2$$ 是将 $$B$$ 按**行**切分的子矩阵。两个部分积分别计算后需要**累加**(all-reduce)。

### 6.2 列并行与行并行

在 Transformer 的一个 block 中, 张量并行的切分策略:

**列并行 (Column-wise TP)**:
- 适用: QKV 投影、FFN 的 up-projection
- 方式: 将权重矩阵按列切分到各 GPU
- 前向: $$y = x [W_1, W_2]$$, 每个 GPU 计算一部分输出, 拼接即可(无需通信)
- 反向: 每个 GPU 有部分输出梯度, 计算 $$x$$ 的梯度时需要 all-reduce 求和

**行并行 (Row-wise TP)**:
- 适用: Attention 输出投影、FFN 的 down-projection
- 方式: 将权重矩阵按行切分到各 GPU
- 前向: $$y = [x_1, x_2] \begin{bmatrix} W_1 \\ W_2 \end{bmatrix} = x_1 W_1 + x_2 W_2$$, 需要 all-reduce 求和
- 反向: 每个 GPU 独立计算自己的 $$W_i$$ 梯度(无需通信)

**关键通信模式**:

| 操作类型 | 前向通信 | 反向通信 |
|:---|:---|:---|
| 列并行 | 无 (identity) | All-Reduce |
| 行并行 | All-Reduce | 无 (identity) |

这个对偶性意味着: **每一对(列并行, 行并行)恰恰完成一次完整的 all-reduce 通信**——列并行的反向 all-reduce 与行并行的前向 all-reduce 各承担一半。

### 6.3 Transformer Block 的 TP 切分全貌

```
输入 x
  ↓
LayerNorm (复制——每个 GPU 一份完整副本, 不切分)
  ↓
QKV 投影 (列并行) → 拼接 → Attention
  ↓
Attention 输出投影 (行并行) → All-Reduce → + x (残差)
  ↓
LayerNorm (复制)
  ↓
Gate & Up 投影 (列并行) → SiLU/SwiGLU → Down 投影 (行并行) → All-Reduce → + x
  ↓
输出
```

每个 Transformer block 内有两处 all-reduce: 分别在 attention 输出投影和 FFN 的 down 投影之后。每次 all-reduce 通信 $$2$$(前向) $$+ 2$$(反向) $$= 4$$ 倍激活量($$bsh$$)。

### 6.4 张量并行的通信量分析

对于每个 Transformer block, 张量并行的通信量:

$$
\text{Communication per block} = 8 \cdot b \cdot s \cdot h \cdot \frac{n_\text{devices} - 1}{n_\text{devices}}
$$

而流水线并行的通信量仅为:

$$
\text{Communication per micro-batch} = b \cdot s \cdot h \quad (\text{点对点})
$$

**结论**: 张量并行的通信量是流水线并行的约 8 倍(每个 block 需要 4 次 all-reduce, 每次 all-reduce 的通信量约为数据量的 2 倍)。因此, **张量并行应在低延迟、高带宽的互联上使用(通常在同一节点的 GPU 之间)**。

### 6.5 张量并行的优缺点

| 维度 | 张量并行 | 流水线并行 |
|:---|:---|:---|
| 气泡 | 无 | 有 ($$\frac{n-1}{n_\text{micro}}$$) |
| 复杂度 | 低 (简单的 wrapper 即可) | 高 (需要调度引擎) |
| 批大小要求 | 无最低要求 | 需要大 $$n_\text{micro}$$ |
| 通信量 | 高 (all-reduce × 4/block) | 低 (点对点 $$bsh$$) |
| 最佳使用场景 | 节点内高速链路 | 跨节点慢速链路 |

### 6.6 实践中 TP 的尺度限制

在所有主流的大模型训练中, **TP 几乎从不超出 8**(一个 DGX 节点的 GPU 数量)。原因是:

1. 超出 8 后, all-reduce 需要跨节点, 通信延迟剧增
2. TP 的通信量与 GPU 数量呈 $$\frac{n-1}{n}$$ 增长(接近饱和), 但跨节点的带宽远低于节点内
3. 8 路 TP 已足够将大部分模型的每层权重切分到足以放入单 GPU 显存

## 七、序列并行与专家并行

### 7.1 激活内存的挑战

回顾激活内存的构成(每层, 存储所有中间结果用于反向传播):

$$
\text{Activation Memory per layer} \approx s \cdot b \cdot h \cdot \left(34 + 5 \cdot \frac{a \cdot s}{h}\right)
$$

其中 $$a$$ 是 attention head 数量, $$s$$ 是序列长度, $$b$$ 是批大小, $$h$$ 是隐藏维度。

这 5 项来自 attention 的二次项(attention scores + dropout masks)。正如 FlashAttention 的意义所示, 通过重计算可以消除这 5 项。但剩余的 34 项中:

- **10 项**来自 LayerNorm(4sbh)、Dropout(2sbh)和 attention/MLP 的输入(4sbh)
- **24 项**来自矩阵乘法的输入(已被张量并行切分)

### 7.2 序列并行的动机

张量并行将矩阵乘法的激活内存减少了 $$N_\text{TP}$$ 倍, 但那 10 项逐点操作(LayerNorm/Dropout)的激活**仍然完整保留**。当序列长度增大时, 这 10sbh 项可以轻易达到 GB 级。

**核心观察**: LayerNorm、Dropout 等操作是**沿序列维的逐点操作**——它们不涉及跨序列的交互。因此, 可以将它们沿序列维切分到不同 GPU 上:

- 前向: 各 GPU 对序列的一部分执行 LayerNorm/Dropout, 然后 all-gather 拼接
- 反向: reduce-scatter 将梯度分散到各 GPU

### 7.3 序列并行与张量并行的组合

在 Megatron-LM 风格的实现中, 序列并行与张量并行**同时使用**:

```
输入 [s, h]
  ↓
All-Gather (序列维): 每个 GPU 获得完整序列
  ↓
矩阵乘法 (TP 切分后的 QKV/FFN)
  ↓
Reduce-Scatter (序列维): 梯度分散
  ↓
LayerNorm (SP 切分后的序列子集)
  ↓
All-Gather → 下一个矩阵乘法 → Reduce-Scatter → ...
```

通过组合 TP + SP, **激活内存(包括逐点操作部分)可以随 GPU 数量线性缩放**。这是实现"显存线性缩放"的最后一块拼图。

### 7.4 专家并行 (Expert Parallelism, EP)

对于 MoE(Mixture of Experts)模型, 专家并行为 MLP 层提供了比张量并行更高效的切分方式。

**工作原理**:

1. 每个专家(一个完整的 MLP)被完整地分配到一个或一组 GPU 上
2. 门控网络 (router) 计算每个 token 应路由到哪些专家 (通常是 top-$$k$$)
3. Token 通过 **all-to-all 通信**被发送到对应专家的 GPU
4. 专家计算完成后, token 通过 all-to-all 返回原 GPU

**通信模式**: 专家并行的通信是 **all-to-all**(而非 TP 的 all-reduce)。All-to-all 是一种更灵活的通信模式——每个 GPU 向每个其他 GPU 发送不同数量的数据——但效率也受通信拓扑影响更大。

### 7.5 EP vs TP 的选择

| 维度 | 专家并行 (EP) | 张量并行 (TP) |
|:---|:---|:---|
| 切分方式 | 按专家路由 token | 按矩阵维切分 |
| 通信 | All-to-all (非结构化) | All-Reduce (结构化) |
| 计算效率 | 专家保持完整(无 matmul 切分) | 矩阵被切分, 可能降低 Tensor Core 效率 |
| 激活节省 | 天然减少 (每 GPU 只处理部分 token) | 需结合 SP |
| 最佳 GPU 数 | 可扩展到很大 (DeepSeek-V3: EP=64) | 通常 ≤ 8 |

**经验法则**: 对于 MoE 模型的 MLP 层, 优先使用 EP(保持专家完整, 避免矩阵切分的效率损失); 对于 Attention 层, 使用 TP(Attention 无法用 EP 切分)。

### 7.6 Attention 与 MLP 的并行解耦

MoE 模型的一个关键架构特征: **MoE 只应用于 MLP 层**(attention 保持稠密)。这导致 attention 和 MLP 对并行策略有不同需求:

- **Attention**: 只能使用 TP 或 DP(无法使用 EP) → 需要较高的 TP
- **MLP**: 优选 EP(如果适用) → 可以降低 TP, 用更多 GPU 做 EP

Megatron-Core 的最新实现支持**分离的并行策略**: 为 attention 和 MLP 分别配置 TP/CP/DP(attention)和 EP/ETP/EDP(MLP)。这提供了更细粒度的控制, 避免了"统一 TP 度"导致的资源浪费。

### 7.7 上下文并行 (Context Parallelism) / Ring Attention

对于**超长上下文**训练, 即使 FlashAttention 也需要将序列沿序列维切分到多个 GPU 上。Ring Attention 的核心思想:

- 将 $$Q, K, V$$ 沿序列维分片到不同 GPU
- 在环形拓扑上轮流传递 $$K, V$$ 的块, 每个 GPU 使用本地 $$Q$$ 与传入的 $$K, V$$ 块计算部分 attention
- 使用 online softmax 在环形遍历过程中维护正确的归一化

这是序列并行的 attention 版——处理的是 attention 的二次复杂度而非 LayerNorm 的线性复杂度。

## 八、三维并行与组合策略

### 8.1 为什么需要混合并行?

没有任何一种单一的并行策略能同时满足所有需求:

| 策略 | 显存缩放 | 通信模式 | 批大小依赖 | 最佳范围 |
|:---|:---|:---|:---|:---|
| DP/ZeRO | 好 (参数) / 差 (激活) | All-Reduce (params) | 是 | 任意规模 |
| TP | 好 (参数 + 激活) | All-Reduce (activations) | 否 | ≤ 8 GPU |
| PP | 好 (参数) | P2P (activations) | 是 | 任意规模 |
| SP | 好 (逐点激活) | All-Gather/Reduce-Scatter | 否 | 与 TP 配合 |
| EP | 好 (专家参数) | All-to-All (tokens) | 是 | 专家数量限制 |

**组合使用**可以扬长避短: 在节点内用 TP(利用高速 NVLink), 跨节点用 PP(利用低通信量), 其余都用 DP 填充。

### 8.2 文献中的经验法则 (Rules of Thumb)

来自 Megatron-LM 团队 (Narayanan et al., 2021) 的实践指南:

**阶段 1 — 让模型放进显存**:
1. 先使用 TP(张量并行)达到每节点的 GPU 数(通常 8)
2. 再使用 PP(流水线并行)跨节点扩展
3. (或者使用 ZeRO-3/FSDP, 取决于带宽条件)

**阶段 2 — 填充剩余 GPU**:
4. 其余所有 GPU 用于 DP(数据并行)

**特殊情况**:
- 如果批大小太小, 使用**梯度累积**来模拟更大的批大小, 以提高通信效率
- 在 MoE 模型中, EP 替代 TP 用于 MLP 层

### 8.3 3D 并行的通信模式总览

```
    数据并行 (DP)
    ↓
    +-- GPU 0-0 ---- GPU 0-1 ---- ... ---- GPU 0-N  (TP 组内: all-reduce)
    |      |             |                      |
    |   GPU 1-0 ---- GPU 1-1 ---- ... ---- GPU 1-N
    |      |             |                      |
    PP  (PP 组间: 点对点激活传递)
    |      |             |                      |
    |   GPU P-0 ---- GPU P-1 ---- ... ---- GPU P-N
    +--
```

在一个典型的 3D 并行拓扑中:
- **TP 组**内部的 GPU 通过 **all-reduce** 频繁通信(每层 4 次)
- **PP 组**之间通过**点对点**传递激活(每微批次 1 次)
- **DP 组**之间通过**all-reduce** 同步梯度(每训练步 1 次)

### 8.4 Narayanan 2021 的缩放策略数据

Megatron-LM 在训练 1T 参数模型时的并行配置:

| 模型规模 | TP | PP | DP |
|:---|:---:|:---:|:---:|
| 1.7 B | 1 | 1 | 32 |
| 3.6 B | 2 | 2 | 32 |
| 7.5 B | 4 | 4 | 32 |
| 18 B | 8 | 4 | 32 |
| 39 B | 8 | 8 | 24 |
| 76 B | 8 | 16 | 15 |
| 145 B | 8 | 32 | 9 |
| 310 B | 8 | 64 | 6 |

**可观察到的模式**:
- TP 首先增加到 8, 然后封顶
- PP 持续增加以容纳更大的模型
- DP 随模型规模增加而逐渐减少(因为更多 GPU 被 TP/PP 占用)

### 8.5 保持线性缩放的关键: 平衡 TP × PP

当 TP=8 固定后, 如何在 PP 和 DP 之间分配更多 GPU?

对 64 节点(512 GPU)的配置实验表明: **TP=8, PP=8 的配置(8×8 网格)通常是最优的**。这反映了 TP 的高效通信(但仅限于小规模)与 PP 的灵活扩展(但需要大微批次)之间的最优平衡点。

### 8.6 激活重计算的额外收益

激活重计算(Activation Recomputation / Gradient Checkpointing)不仅节省显存——它还能**间接改善吞吐量**:

通过重计算释放的显存可以用于增大微批次大小, 从而:
- 对于 PP: 更大的 $$n_\text{micro}$$ 意味着更小的气泡比例
- 对于 DP: 更大的有效批大小提升了通信效率
- 对于 TP: 更大的矩阵乘法提升了 Tensor Core 利用率

在 Megatron 的 benchmarks 中(TP=8, PP=16), 启用选择性重计算可以将吞吐量提升 30% 以上——不是因为重计算本身更高效, 而是因为它释放的显存允许使用更大的微批次。

## 九、真实系统案例分析

### 9.1 主流模型的并行配置总览

| 模型 | DP | TP/SP | EP | PP | CP | 关键模式 |
|:---|:---:|:---:|:---:|:---:|:---:|:---|
| **DeepSeek-V2** | ? (ZeRO-1) | 1 | 8 | 16 | ? | EP + PP 为主 |
| **DeepSeek-V3** | ? (ZeRO-1) | 1 | 64 | 16 | ? | EP 扩展到 64 路, 8 节点 |
| **Llama 3 405B** | 128 | 8 | 0 | 16 | 1 | 经典 TP+PP+DP, 无 MoE |
| **Gemma 2 (27B)** | 768 | 8 | 0 | 0 | 0 | 纯 ZeRO-3 + TP |
| **Mixtral 8×22B** | 2 | 4 | 8 | 4 | 1 | TP+EP+PP 组合 |
| **Nemotron 3 120B** | ? | 2 | 64 | ? | 64 | 长上下文: 大量 CP |
| **Qwen 3 235B-A22B** | ? | 2 | 32 | 8 | 1 | EP 为主 (8 GPU 内) |

### 9.2 深度案例分析: DeepSeek-V3

DeepSeek-V3 是目前最大规模的 MoE 训练系统之一, 其并行配置展示了极致的系统优化:

- **PP=16**: 16 路流水线并行, 按深度切分 61 层 Transformer
- **EP=64(8 节点)**: 64 路专家并行, 将 256 个细粒度专家分布在 8 个节点上
- **ZeRO-1**: 仅分片 optimizer state(通信量最少)
- **EP 的 1F1B All-to-All 重叠**: 将 all-to-all 通信与专家计算的 1F1B 调度重叠, 隐藏通信延迟

**关键洞察**: DeepSeek-V3 几乎不使用 TP(TP=1), 而是大量使用 EP。这是因为对于 MoE 模型, EP 在 MLP 层的通信效率(路由 token 而非切分矩阵)和计算效率(专家保持完整)都优于 TP。

### 9.3 深度案例分析: Llama 3 405B

Llama 3 405B 是最大的稠密模型之一(无 MoE):

- **DP=128, TP=8, PP=16**: 总计 128 × 8 × 16 = 16384 GPU
- 第一阶段(小批量训练): 使用较小的全局批大小, 更依赖 PP+TP
- 第二阶段(预训练): 全局批大小 ~16M tokens
- 第三阶段(长上下文扩展): 切换到更大的 CP 度

**大规模训练的运维现实**: Llama 3 团队报告在训练过程中遇到了大量 GPU 故障——16K GPU 集群上每 1-3 小时就有一块 GPU 出现硬件或软件故障。这需要复杂的检查点/恢复机制和动态 GPU 替换策略。

### 9.4 深度案例分析: Qwen 3

Qwen 3 代表了 MoE 模型的最新实践:

- 较小变体(30B-A3B): 主要在 8 GPU 内使用 EP
- 较大变体(235B-A22B): 2/8/32 的 TP/PP/EP 配置

**趋势**: 从 Yi-lightning(2025)开始, 新模型倾向于用 EP 替代 TP 用于 MoE 的 MLP 层。这反映了硬件演进(更好的 all-to-all 支持)和算法优化(EP 路由策略的成熟)。

### 9.5 关键模式总结

从这些真实系统中可以提炼出以下模式:

1. **TP ≤ 8**: 几乎所有模型都将 TP 限制在 8 以内(一个 DGX 节点), 对应的正是 NVLink 域的大小。
2. **EP 可以很大但很难调**: DeepSeek-V3 的 EP=64, Nemotron 的 EP=64——但 EP 需要处理负载均衡和通信调度。
3. **长上下文需要大量 CP**: Nemotron 和 Llama 3 在长上下文阶段使用了 64 路 CP。
4. **ZeRO-1 是安全低成本的默认选择**: DeepSeek、Yi 等模型都使用 ZeRO-1 作为 DP 的优化器——通信量与朴素 DDP 相同, 但 optimizer state 显存减少 N 倍。
5. **PP 仍然重要**: 尽管有气泡, PP 的低通信特性使其成为跨节点扩展的主要方式。几乎所有最大的模型(DeepSeek-V3, Llama 3 405B)都使用 PP=16。

---

## 十、总结

### 10.1 并行策略选择决策树

```
模型规模 > 单 GPU 显存?
  ├── 否 → 朴素 DDP + ZeRO-1 (免费 optimizer state 分片)
  └── 是 →
      ├── 首先, 在节点内用 TP (≤ 8) — 利用 NVLink 高速互联
      ├── 其次, 跨节点用 PP — 利用低通信量 (点对点 $$bsh$$)
      ├── 如果是 MoE 模型, MLP 层用 EP 替代 TP
      ├── 对于长上下文, 添加 CP 或 SP
      └── 剩余的 GPU 全部用于 DP (配合 ZeRO-1 或 ZeRO-3)
```

### 10.2 核心要点

1. **分布式训练的本质**: 在以**通信带宽**换取**显存容量**和**计算吞吐量**。每个并行策略代表了这个权衡空间中的不同最优点。

2. **ZeRO 的优雅性**: 通过将 all-reduce 分解为 reduce-scatter + all-gather, ZeRO 实现了显存的递减式节省, 而 ZeRO-1(甚至 ZeRO-2)在带宽受限时通信成本完全不变。

3. **TP vs PP 的选择**: TP 无气泡但通信量大(适合节点内 NVLink), PP 有气泡但通信量小(适合跨节点 InfiniBand)。两者互补, 几乎总是同时使用。

4. **没有银弹**: DeepSeek-V3 用 EP=64 + PP=16, Llama 3 用 TP=8 + PP=16 + DP=128——最优配置取决于模型架构(MoE vs 稠密)、集群拓扑和训练规模。

5. **新的单位是数据中心**: 在现代 LLM 训练中, 我们不再以单个 GPU 为思考单位, 而是以整个数据中心(含网络拓扑)为计算单元。并行策略的设计是对这个数据中心级"计算机"的系统编程。

### 10.3 与课程整体的连接

本章在 CS 336 课程体系中处于关键位置:
- **前序章节**: 第 5 章建立了 GPU 硬件和通信链路的理解, 第 6 章教会了单 GPU 上的高效内核编程
- **本章**: 将单 GPU 的计算高效性扩展到多 GPU 规模——回答"当模型大到放不进一个 GPU 时怎么办"
- **后续章节**: 第 8 章将进入训练稳定性、超参数调优和规模化定律——这些主题的讨论依赖于本章的并行系统视角(例如, 批大小对流水线效率的影响)

理解并行策略不仅是"工程细节"——它直接影响你能训练多大规模的模型、使用什么样的架构(MoE vs 稠密), 以及能以多大的批大小进行训练。所有这些反过来又影响着损失曲线的形状和最终模型的质量。
